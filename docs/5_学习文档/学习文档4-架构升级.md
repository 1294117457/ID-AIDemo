> 前提：不修改 idbackend / idfrontend / idfrontend-admin，只改 ID-AIDemo 工程内部。

```
│   ├── state.ts

│   └── knowledgeManager.ts   ← MemoryVectorStore，重启丢数据

```

### 当前存在的问题

1. **mainGraph.ts 职责过重**：图定义 + HTTP 适配（invokeAgent/streamAgent/extractResult/checkInterrupt）全在一个 248 行的文件里
2. **Prompt 散落各处**：classifyNode 里 40 行 prompt、applyGraph 里写死 prompt、consultGraph 里又有一套，改一个要翻三个文件
3. **LLM 创建重复**：`new ChatOpenAI({ apiKey: getApiKey(), ... })` 在 mainGraph、applyGraph、consultGraph、analyze.ts 里写了 4 遍
4. **multer/UPLOAD_DIR 重复**：agent.ts、analyze.ts、knowledge.ts 三个路由文件各自创建一遍
5. **知识库/对话不持久**：MemoryVectorStore 重启丢数据，MemorySaver 重启丢会话
6. **aiConfig 过于复杂**：为了"在线改配置"引入了整套 DB 读写 + 缓存 + TTL

```typescript
  return `分析以下用户的多轮输入，判断意图并提取信息：

【分类与校验规则】：
1. 如果用户是单纯询问政策或了解相关信息（如"挑战杯能加多少分"），意图记为 consult。
2. 如果用户表达了"想申请加分"或正在提交比赛凭证，必须严格校验其提供的信息是否完整！
   完整材料必须同时包含以下4个关键要素：【赛事名称】、【奖项等级】、【时间】、【申请人角色】。

${allUserText}`
```

```typescript
// src/agent/graphs/mainGraph.ts
import { StateGraph, START, END, interrupt, MemorySaver } from '@langchain/langgraph'
import { HumanMessage, AIMessage } from '@langchain/core/messages'
import { z } from 'zod'
import { MainState, MainStateType } from '../state.js'
import { applySubgraph } from './applyGraph.js'
import { consultSubgraph } from './consultGraph.js'
import { createChatModel } from '../../services/llmService.js'
import { classifyPrompt, askForMorePrompt } from '../prompts.js'

const checkpointer = new MemorySaver()

async function classifyNode(state: MainStateType) {
  const allUserText = state.messages
    .filter(m => m instanceof HumanMessage)
    .map(m => m.content).join('\n')

  const reply = await createChatModel(0)
    .withStructuredOutput(z.object({ /* schema */ }))
    .invoke([new HumanMessage(classifyPrompt(allUserText))])

  return { intent: reply.intent, missingInfo: reply.missing || [], documentText: reply.documentText || '' }
}

async function askForMoreNode(state: MainStateType) {
  const question = askForMorePrompt(state.missingInfo)
  const userAnswer = interrupt(question)
  return { messages: [new AIMessage(question), new HumanMessage(String(userAnswer))] }
}

const mainGraph = new StateGraph(MainState)
  .addNode('classify', classifyNode)
  .addNode('ask', askForMoreNode)
  .addNode('applyGraph', applySubgraph)
  .addNode('consultGraph', consultSubgraph)
  .addEdge(START, 'classify')
  .addConditionalEdges('classify', (s) => s.intent, {
    insufficient: 'ask', apply: 'applyGraph', consult: 'consultGraph'
  })
  .addEdge('ask', 'classify')
  .addEdge('applyGraph', END)
  .addEdge('consultGraph', END)

export function getCompiledGraph() {
  return mainGraph.compile({ checkpointer })
}
```

## 阶段 4：对话持久化（SqliteSaver）

> 目标：服务重启后对话不丢失

### 4.1 原理

当前用 `MemorySaver`，所有 checkpointer 数据存在内存里，重启就没了。
LangGraph 自带 `SqliteSaver`（@langchain/langgraph-checkpoint-sqlite），把对话状态写入 SQLite。

### 4.2 安装依赖

```bash
npm install @langchain/langgraph-checkpoint-sqlite
```

### 4.3 改造 mainGraph.ts

```typescript
// 改前：
import { MemorySaver } from '@langchain/langgraph'
const checkpointer = new MemorySaver()

// 改后：
import { SqliteSaver } from '@langchain/langgraph-checkpoint-sqlite'
import path from 'path'
import { fileURLToPath } from 'url'

const __dirname = path.dirname(fileURLToPath(import.meta.url))
const CHECKPOINT_PATH = path.resolve(__dirname, '../../../data/checkpoints.db')

// SqliteSaver.fromConnString 返回 Promise，需要异步初始化
let checkpointer: SqliteSaver

export async function getCompiledGraph() {
  if (!checkpointer) {
    checkpointer = SqliteSaver.fromConnString(CHECKPOINT_PATH)
  }
  return mainGraph.compile({ checkpointer })
}
```

因为 `getCompiledGraph()` 变成了异步函数，`agentService.ts` 也要对应调整：

## 阶段 5：知识库持久化（FAISS）

> 目标：服务重启后知识库不需要重新 embedding，省时间省钱

### 5.1 原理

当前 `MemoryVectorStore` 纯内存，每次启动都要重新 embedding 所有文件（调 API，消耗 token）。
换成 `FaissStore`，向量存在磁盘文件里，启动时直接加载。

### 5.2 安装依赖

```bash
npm install faiss-node
```

（你的 package.json 里已经有 faiss-node，确认一下就行）

### 5.3 改造 knowledgeManager.ts（重命名为 knowledgeService.ts 可选）

核心改动是把 `MemoryVectorStore` 换成 `FaissStore`，加上 `save()` / `load()` 逻辑：

```typescript
import { FaissStore } from '@langchain/community/vectorstores/faiss'

const FAISS_DIR = path.resolve(__dirname, '../../data/faiss_index')

let vectorStore: FaissStore | null = null

async function getVectorStore() {
  if (!vectorStore) {
    // 尝试从磁盘加载已有索引
    if (fs.existsSync(path.join(FAISS_DIR, 'faiss.index'))) {
      vectorStore = await FaissStore.load(FAISS_DIR, createEmbeddings())
      console.log('[knowledge] 从磁盘加载 FAISS 索引')
    } else {
      vectorStore = new FaissStore(createEmbeddings(), {})
    }
  }
  return vectorStore
}

/** 每次修改后保存到磁盘 */
async function saveIndex() {
  if (vectorStore) {
    fs.mkdirSync(FAISS_DIR, { recursive: true })
    await vectorStore.save(FAISS_DIR)
  }
}
```

然后在 `ingestFile`、`removeSource` 等修改操作之后调用 `saveIndex()`。

`initKnowledge` 改为：如果 FAISS 索引已存在就直接加载（秒级），不存在才走 embedding 入库流程。

### 5.4 验证

1. 第一次启动：看到 `[knowledge] ✓ xxx.pdf → N 块`（正常 embedding）
2. 第二次启动：看到 `[knowledge] 从磁盘加载 FAISS 索引`（没有重新 embedding，启动快很多）
3. 上传新文件 → 确认能正常入库 + 搜索

## 阶段 7：增强 HITL（Human-in-the-Loop）

> 目标：在 apply 流程中增加"确认提交"步骤

### 7.1 当前 HITL 流程

```
用户消息 → classify → insufficient? → ask(interrupt) → 用户补充 → classify → ...
```

只在 **信息不完整** 时 interrupt。材料匹配完直接返回结果，用户没有确认环节。

### 7.2 升级后 HITL 流程

```
用户消息 → classify → insufficient? → ask(interrupt) → 用户补充 → classify
                                                                      ↓ apply
                                                               fetchPolicy
                                                                      ↓
                                                            analyzeAndMatch
                                                                      ↓
                                                               summarize
                                                                      ↓
                                                          confirm(interrupt) ← 【新增】
                                                                      ↓
                                                          用户确认"是/否"
                                                                      ↓ 是
                                                               submitNode ← 【新增】
```

### 7.3 实现步骤

**在 applyGraph.ts 中加两个节点：**

```typescript
import { interrupt } from '@langchain/langgraph'

// summarize 之后新增 confirmNode
async function confirmNode(state: ApplyStateType) {
  const suggestions = state.checkResults
    .map(r => { try { return JSON.parse(r) } catch { return null } })
    .filter(Boolean)

  if (suggestions.length === 0 || suggestions[0]?.error) {
    // 没有匹配项，不需要确认
    return {}
  }

  const summary = suggestions.map((s: any) =>
    `• ${s.templateName} / ${s.ruleName}（预计 ${s.estimatedScore} 分）`
  ).join('\n')

  // interrupt 暂停，等用户确认
  const userConfirm = interrupt(
    `为您匹配到以下加分项：\n${summary}\n\n确认提交申请吗？（回复"确认"继续，其他内容取消）`
  )

  return { userConfirm: String(userConfirm) }
}

// 条件路由：用户确认 → submit，否则 → end
function confirmRouter(state: ApplyStateType): string {
  const answer = (state as any).userConfirm ?? ''
  return answer.includes('确认') ? 'submit' : 'cancel'
}

async function submitNode(state: ApplyStateType) {
  return { messages: [new AIMessage('已为您提交加分申请，请在"我的申请"中查看进度。')] }
}

async function cancelNode(state: ApplyStateType) {
  return { messages: [new AIMessage('已取消本次申请。如需重新申请，请再次发送材料信息。')] }
}
```

**修改图结构：**

