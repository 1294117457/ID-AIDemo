### 第二步：写 SQLite 建表

**练习：** 创建 `src/db.ts`，初始化 SQLite，建一张存向量数据的表。

想想这张表需要哪些字段：
- 文件名（这段文本来自哪个文件）
- 块序号（是第几块）
- 文本内容
- 向量数据（embedding，是一个浮点数组）

试着自己写，写完看答案。

**答案：**

```typescript
// src/db.ts
import Database from 'better-sqlite3'
import { mkdirSync } from 'fs'

mkdirSync('data', { recursive: true })
const db = new Database('data/agent.db')

db.exec(`
  CREATE TABLE IF NOT EXISTS knowledge_chunks (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    source_file TEXT    NOT NULL,
    chunk_index INTEGER NOT NULL,
    content     TEXT    NOT NULL,
    embedding   TEXT    NOT NULL,
    created_at  INTEGER NOT NULL DEFAULT (unixepoch())
  );
  CREATE INDEX IF NOT EXISTS idx_source ON knowledge_chunks(source_file);
`)

export default db
```

**关键点：** `embedding` 字段是 `TEXT` 类型，不是什么向量专属类型。SQLite 没有原生向量类型，所以我们把 `number[]` 用 `JSON.stringify` 变成字符串存进去。这个项目数据量不大（几百个 chunk），暴力搜索足够了，不需要专门的向量数据库。

### 第三步：写文本分块函数

**练习：** 创建 `src/chunk.ts`，写一个 `chunkText(text, chunkSize, overlap)` 函数。

规则：
- 把长文本按 `chunkSize`（默认 500 字符）切成小块
- 每块之间有 `overlap`（默认 100 字符）的重叠
- 太短的块（< 30 字符）扔掉

比如一段 1200 字的文本，chunkSize=500, overlap=100：
- 块 0：第 0~499 字符
- 块 1：第 400~899 字符（往前退了 100 个字符的 overlap）
- 块 2：第 800~1199 字符

**答案：**

```typescript
// src/chunk.ts
export function chunkText(text: string, chunkSize = 500, overlap = 100): string[] {
  const chunks: string[] = []
  const clean = text.replace(/\r\n/g, '\n').replace(/\n{3,}/g, '\n\n').trim()

  let start = 0
  while (start < clean.length) {
    const end = Math.min(start + chunkSize, clean.length)
    const chunk = clean.slice(start, end).trim()
    if (chunk.length >= 30) chunks.push(chunk)
    if (end === clean.length) break
    start += chunkSize - overlap   // 步进 = chunkSize - overlap = 400
  }
  return chunks
}
```

**为什么要 overlap？** 假设一段关键信息刚好跨越了两个块的边界，比如"挑战杯国赛一等奖"这几个字被分成了"挑战杯国赛一"和"等奖"。有 overlap 的话，两个相邻块都会包含完整的这段话，搜索时就不会漏掉。

### 第四步：写 Embedding 调用

**练习：** 创建 `src/embedding.ts`，调 DashScope 的 embedding API，把文本变成向量。

API 格式和 OpenAI 一样：

```
POST https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings
Header: Authorization: Bearer sk-xxx
Body: { "model": "text-embedding-v3", "input": ["你的文本"] }
返回: { "data": [{ "index": 0, "embedding": [0.012, -0.034, ...] }] }
```

注意两点：
- 每条文本最多 512 字符，超了要截断
- 批量调用时每批最多 6 条（不然容易超 token 限制）

**答案：**

```typescript
// src/embedding.ts
import 'dotenv/config'
import axios from 'axios'

const API_KEY = process.env.QWEN3_API_KEY!
const BASE_URL = 'https://dashscope.aliyuncs.com/compatible-mode/v1'
const MODEL = 'text-embedding-v3'

export async function getEmbedding(text: string): Promise<number[]> {
  const resp = await axios.post(
    `${BASE_URL}/embeddings`,
    { model: MODEL, input: [text.slice(0, 512)] },
    { headers: { Authorization: `Bearer ${API_KEY}` }, timeout: 30000 }
  )
  return resp.data.data[0].embedding
}

export async function getEmbeddings(texts: string[]): Promise<number[][]> {
  const results: number[][] = []
  for (let i = 0; i < texts.length; i += 6) {
    const batch = texts.slice(i, i + 6).map(t => t.slice(0, 512))
    const resp = await axios.post(
      `${BASE_URL}/embeddings`,
      { model: MODEL, input: batch },
      { headers: { Authorization: `Bearer ${API_KEY}` }, timeout: 60000 }
    )
    const sorted = resp.data.data.sort((a: any, b: any) => a.index - b.index)
    results.push(...sorted.map((d: any) => d.embedding))
  }
  return results
}
```

**这里返回的 `number[]` 是什么？** 就是一个大约 1024 个浮点数组成的数组，比如 `[0.012, -0.034, 0.056, ...]`。两段语义相似的文本，它们的向量在 1024 维空间里的方向会比较接近。

### 第五步：写向量存储和搜索

**练习：** 创建 `src/vectorStore.ts`，实现：
- `saveChunk`：存一条向量数据
- `searchSimilar`：给一个查询向量，找出最相似的 topK 条

搜索的核心是**余弦相似度**。两个向量 A 和 B：

```
cosine(A, B) = (A·B) / (|A| × |B|)
```

就是两个向量的夹角的余弦值。值越接近 1，说明方向越一致，语义越相似。

**答案：**

```typescript
// src/vectorStore.ts
import db from './db.js'

function cosineSimilarity(a: number[], b: number[]): number {
  let dot = 0, normA = 0, normB = 0
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i]
    normA += a[i] ** 2
    normB += b[i] ** 2
  }
  const denom = Math.sqrt(normA) * Math.sqrt(normB)
  return denom === 0 ? 0 : dot / denom
}

export function saveChunk(sourceFile: string, chunkIndex: number, content: string, embedding: number[]) {
  db.prepare(
    'INSERT INTO knowledge_chunks (source_file, chunk_index, content, embedding) VALUES (?, ?, ?, ?)'
  ).run(sourceFile, chunkIndex, content, JSON.stringify(embedding))
}

export function searchSimilar(queryEmbedding: number[], topK = 5) {
  const rows = db.prepare('SELECT id, source_file, content, embedding FROM knowledge_chunks').all() as any[]

  return rows
    .map(r => ({
      id: r.id,
      sourceFile: r.source_file,
      content: r.content,
      similarity: cosineSimilarity(queryEmbedding, JSON.parse(r.embedding))
    }))
    .sort((a, b) => b.similarity - a.similarity)
    .slice(0, topK)
}
```

**这个搜索方式效率怎么样？** 说实话很暴力——每次查询都把所有向量从数据库读出来，逐个算余弦相似度。但对于几百条 chunk 来说完全没问题。如果数据量上万，就该用 pgvector 或 Milvus 这种带索引的向量数据库了。

### 第二步：写文档解析器

**练习：** 创建 `src/docParser.ts`，写一个 `parseFile(filePath, ext)` 函数，根据文件扩展名选择不同的解析方式。

提示：
- PDF 用 `pdf-parse`
- DOCX 用 `mammoth`
- XLSX 用 `xlsx`
- TXT/MD 直接读文件

**答案：**

```typescript
// src/docParser.ts
import fs from 'fs'
import path from 'path'

export async function parseFile(filePath: string, hintExt?: string): Promise<string> {
  const ext = (hintExt ?? path.extname(filePath)).toLowerCase()
  switch (ext) {
    case '.pdf':  return parsePdf(filePath)
    case '.docx':
    case '.doc':  return parseDocx(filePath)
    case '.xlsx':
    case '.xls':  return parseXlsx(filePath)
    case '.md':
    case '.txt':  return fs.readFileSync(filePath, 'utf-8')
    default:      return ''
  }
}

async function parsePdf(filePath: string): Promise<string> {
  const { PDFParse } = await import('pdf-parse')
  const buffer = fs.readFileSync(filePath)
  const parser = new PDFParse({ data: buffer })
  await (parser as any).load()
  const result = await parser.getText()
  return result.text
}

async function parseDocx(filePath: string): Promise<string> {
  const mammoth = await import('mammoth')
  const result = await mammoth.extractRawText({ path: filePath })
  return result.value
}

async function parseXlsx(filePath: string): Promise<string> {
  const XLSX = await import('xlsx')
  const readFile = (XLSX.default?.readFile ?? XLSX.readFile) as Function
  const workbook = readFile(filePath)
  const lines: string[] = []
  for (const sheetName of workbook.SheetNames) {
    const rows = XLSX.utils.sheet_to_json(workbook.Sheets[sheetName], { header: 1 }) as any[][]
    lines.push(`【${sheetName}】`)
    for (const row of rows) {
      const line = row.map((c: any) => c == null ? '' : String(c)).join('\t').trim()
      if (line) lines.push(line)
    }
  }
  return lines.join('\n')
}
```

**为什么用 `await import()` 动态导入？** 因为 `pdf-parse` 这个库在 Node ESM 下直接 import 会有兼容性问题，动态导入可以绕过。正式工程的 `docParser.ts` 还加了一堆 polyfill（DOMMatrix、ImageData 之类的），那是因为 pdf-parse 内部依赖了一些浏览器 API，在 Node 里不存在。

### 第三步：写上传接口

**练习：** 创建 `src/server.ts`，提供一个 `POST /knowledge/upload` 接口：
1. 用 multer 接收文件
2. 调 parseFile 解析成文本
3. 调 chunkText 分块
4. 调 getEmbeddings 向量化
5. 存入数据库
6. 删除临时文件

**答案：**

```typescript
// src/server.ts
import 'dotenv/config'
import './db.js'
import express from 'express'
import cors from 'cors'
import multer from 'multer'
import path from 'path'
import fs from 'fs'
import { parseFile } from './docParser.js'
import { chunkText } from './chunk.js'
import { getEmbeddings } from './embedding.js'
import { saveChunk, searchSimilar } from './vectorStore.js'
import { getEmbedding } from './embedding.js'
import db from './db.js'

const app = express()
app.use(cors())
app.use(express.json())

fs.mkdirSync('uploads', { recursive: true })
const upload = multer({ dest: 'uploads/' })

// 上传文档到知识库
app.post('/knowledge/upload', upload.single('file'), async (req, res) => {
  if (!req.file) { res.json({ code: 400, msg: '没有文件' }); return }

  const originalName = Buffer.from(req.file.originalname, 'latin1').toString('utf8')
  const ext = path.extname(originalName).toLowerCase()

  try {
    // 解析
    const text = await parseFile(req.file.path, ext)
    if (!text.trim()) {
      res.json({ code: 400, msg: '文件内容为空' }); return
    }
    // 分块
    const chunks = chunkText(text, 500, 100)
    // 向量化
    const embeddings = await getEmbeddings(chunks)
    // 存库（先删旧的同名文件数据）
    db.prepare('DELETE FROM knowledge_chunks WHERE source_file = ?').run(originalName)
    for (let i = 0; i < chunks.length; i++) {
      saveChunk(originalName, i, chunks[i], embeddings[i])
    }

    res.json({ code: 200, msg: '上传成功', data: { fileName: originalName, chunkCount: chunks.length } })
  } catch (err) {
    res.json({ code: 500, msg: `处理失败: ${err}` })
  } finally {
    fs.unlink(req.file.path, () => {})
  }
})

// 搜索知识库
app.post('/knowledge/search', async (req, res) => {
  const { query, topK = 5 } = req.body
  const embedding = await getEmbedding(query)
  const results = searchSimilar(embedding, topK)
  res.json({ code: 200, data: results })
})

// 列出已入库的文件
app.get('/knowledge/list', (_req, res) => {
  const sources = db.prepare(
    'SELECT source_file as sourceFile, COUNT(*) as chunkCount FROM knowledge_chunks GROUP BY source_file'
  ).all()
  res.json({ code: 200, data: sources })
})

app.listen(3001, () => console.log('运行中 → http://localhost:3001'))
```

启动：`npx tsx src/server.ts`

然后用 curl 测试上传：

```bash
curl -F "file=@你的文件.pdf" http://localhost:3001/knowledge/upload
```

## Day 3：用 MCP 把知识库操作封装成工具

今天的目标：理解 MCP 到底干了什么，把 Day 2 直接调用的函数改成通过 MCP 调用。

### MCP 解决什么问题

Day 2 的上传接口里，你直接调用了 `chunkText`、`getEmbeddings`、`saveChunk`。这样做完全能跑，但正式工程没这么干，而是加了一层 MCP。

MCP（Model Context Protocol）的核心思路：**把一组操作封装成"工具"，通过标准协议调用**。

在这个项目里：
- MCP Server 是一个独立的子进程，注册了 search_knowledge、ingest_document 等工具
- MCP Client 在主进程里，需要调知识库时就通过 stdio（标准输入输出）给 Server 发命令
- Agent 的节点不需要关心向量怎么存的，只要 `callTool('search_knowledge', {query})` 就行

好处：
1. Agent 代码和知识库实现解耦
2. 以后换成远程的 MCP Server（比如接一个真正的向量数据库），Agent 代码不用改
3. MCP 是标准协议，其他 Agent 框架也能调用这些工具

### 第一步：安装 MCP 依赖

```bash
npm install @modelcontextprotocol/sdk @langchain/mcp-adapters zod
```

### 第二步：写 MCP Server

**练习：** 创建 `src/mcp/knowledgeServer.ts`，注册以下工具：
- `search_knowledge`：接收 query 和 topK，返回搜索结果
- `ingest_document`：接收 sourceFile 和 text，执行分块+向量化+存库

MCP Server 用 `StdioServerTransport`，就是通过 stdin/stdout 通信。

**答案：**

```typescript
// src/mcp/knowledgeServer.ts
import 'dotenv/config'
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'
import { z } from 'zod'

// 这里直接复用之前写的模块
import '../db.js'
import { getEmbedding, getEmbeddings } from '../embedding.js'
import { searchSimilar, saveChunk } from '../vectorStore.js'
import { chunkText } from '../chunk.js'
import db from '../db.js'

const server = new McpServer({ name: 'knowledge-server', version: '1.0.0' })

// 工具 1：搜索知识库
server.tool(
  'search_knowledge',
  '在知识库中搜索相关内容',
  {
    query: z.string().describe('搜索关键词'),
    topK: z.number().default(5).describe('返回数量')
  },
  async ({ query, topK }) => {
    const embedding = await getEmbedding(query)
    const chunks = searchSimilar(embedding, topK)
    const text = chunks.length > 0
      ? chunks.map((c, i) => `[${i + 1}] (${c.sourceFile}) ${c.content}`).join('\n\n')
      : '（知识库暂无相关内容）'
    return { content: [{ type: 'text' as const, text }] }
  }
)

// 工具 2：导入文档到知识库
server.tool(
  'ingest_document',
  '将文本分块并存入向量知识库',
  {
    sourceFile: z.string().describe('文件名'),
    text: z.string().describe('文档文本'),
  },
  async ({ sourceFile, text }) => {
    // 先删旧数据
    db.prepare('DELETE FROM knowledge_chunks WHERE source_file = ?').run(sourceFile)
    // 分块 + 向量化 + 存储
    const chunks = chunkText(text, 500, 100)
    const embeddings = await getEmbeddings(chunks)
    for (let i = 0; i < chunks.length; i++) {
      saveChunk(sourceFile, i, chunks[i], embeddings[i])
    }
    return { content: [{ type: 'text' as const, text: `已入库：${sourceFile}，共 ${chunks.length} 个分块` }] }
  }
)

// 启动（stdio 模式）
async function main() {
  const transport = new StdioServerTransport()
  await server.connect(transport)
  console.error('[MCP Server] 已启动')   // 注意用 stderr，因为 stdout 被 MCP 协议占了
}
main()
```

**注意：** Server 的日志要用 `console.error` 而不是 `console.log`，因为 stdout 是 MCP 协议的通信通道，你打印的内容会干扰协议数据。

### 第三步：写 MCP Client

**练习：** 创建 `src/mcp/mcpClient.ts`，用 `MultiServerMCPClient` 连接上面的 Server。

Client 要做的事：
1. 启动一个子进程运行 knowledgeServer.ts
2. 通过 stdio 跟它通信
3. 提供 `callTool(name, input)` 函数供外部调用

**答案：**

```typescript
// src/mcp/mcpClient.ts
import { MultiServerMCPClient } from '@langchain/mcp-adapters'
import type { StructuredToolInterface } from '@langchain/core/tools'

let client: MultiServerMCPClient | null = null
let tools: StructuredToolInterface[] | null = null

async function ensureClient() {
  if (client) return client
  client = new MultiServerMCPClient({
    knowledge: {
      transport: 'stdio' as const,
      command: 'node',
      args: ['--import', 'tsx', 'src/mcp/knowledgeServer.ts'],
    },
  })
  tools = await client.getTools()
  console.log(`[MCP Client] 已连接，工具: ${tools.map(t => t.name).join(', ')}`)
  return client
}

export async function callTool(name: string, input: Record<string, any>): Promise<string> {
  const allTools = await (async () => { await ensureClient(); return tools! })()
  const tool = allTools.find(t => t.name === name)
  if (!tool) throw new Error(`MCP tool not found: ${name}`)
  const result = await tool.invoke(input)
  return typeof result === 'string' ? result : JSON.stringify(result)
}

export async function closeMcpClient() {
  if (client) { await client.close(); client = null; tools = null }
}
```

**这里发生了什么？** `ensureClient` 第一次被调用时，会启动一个子进程 `node --import tsx src/mcp/knowledgeServer.ts`。这个子进程就是 MCP Server。然后 Client 通过这个子进程的 stdin/stdout 跟它通信。

`callTool('search_knowledge', {query: '挑战杯', topK: 5})` 实际发生的事：
1. Client 把请求序列化成 JSON，写入子进程的 stdin
2. Server 收到请求，执行 search_knowledge 工具
3. Server 把结果写到 stdout
4. Client 读取 stdout，反序列化得到结果

把 Day 2 的 server.ts 里直接调 chunkText/getEmbeddings 的代码，换成 callTool：

```typescript
for (let i = 0; i < chunks.length; i++) saveChunk(originalName, i, chunks[i], embeddings[i])
```

一行搞定，不用关心内部怎么分块怎么向量化。

同理搜索接口也可以改：


- `src/mcp/knowledgeServer.ts` → 正式工程 `src/mcp/knowledgeServer.ts`（多了 parse_document、list、delete 三个工具）

## Day 4：写 LangGraph Agent 核心（图结构 + 意图分类）

今天是最核心的一天。你要用 LangGraph 写出整个 Agent 的图结构。

### 先搞清楚这个 Agent 要干什么

这个 Agent 有两种能力：
1. **咨询**（consult）：学生问"挑战杯能加多少分"→ 从知识库搜相关政策 → LLM 基于政策回答
2. **申报**（apply）：学生上传证书 → 从知识库搜相关政策 → LLM 匹配加分模板 → 给出建议

所以 Agent 收到消息后第一步是**判断意图**。如果是申报但信息不全（比如没传文件），还要**打断**问用户补充信息。

### 第一步：安装 LangGraph 依赖

```bash
npm install @langchain/langgraph @langchain/openai @langchain/core langchain
```

### 第二步：定义图的 State

**练习：** 创建 `src/agent/state.ts`，用 LangGraph 的 `Annotation` 定义 State。

Agent 运行过程中需要共享的数据有：
- `messages`：对话历史（LangGraph 内置的）
- `intent`：意图分类结果（consult / apply / insufficient）
- `missingInfo`：缺少的信息列表
- `documentText`：上传文档的解析文本
- `retrievedContext`：从知识库检索到的内容
- `checkResults`：申报匹配结果
- `templates`：加分模板数据

**答案：**

```typescript
// src/agent/state.ts
import { MessagesAnnotation, Annotation } from '@langchain/langgraph'

export const MainState = Annotation.Root({
  ...MessagesAnnotation.spec,   // 包含 messages 字段，自动追加消息

  intent: Annotation<'consult' | 'apply' | 'insufficient'>({
    reducer: (_, x) => x,       // 每次直接覆盖
    default: () => 'consult'
  }),
  missingInfo: Annotation<string[]>({
    reducer: (_, x) => x,
    default: () => []
  }),
  documentText: Annotation<string>({
    reducer: (_, x) => x,
    default: () => ''
  }),
  retrievedContext: Annotation<string>({
    reducer: (_, x) => x,
    default: () => ''
  }),
  checkResults: Annotation<string[]>({
    reducer: (old, newVal) => [...(old ?? []), ...(newVal ?? [])],  // 追加模式
    default: () => []
  }),
  templates: Annotation<any[]>({
    reducer: (_, x) => x,
    default: () => []
  }),
  policyContext: Annotation<string>({
    reducer: (_, x) => x,
    default: () => ''
  }),
})

export type MainStateType = typeof MainState.State
```

**`reducer` 是什么意思？** 当一个节点返回 `{ intent: 'consult' }` 时，LangGraph 用 reducer 函数来决定怎么更新 state。`(_, x) => x` 就是直接替换。`checkResults` 的 reducer 是追加模式，这样多个匹配结果不会互相覆盖。

### 第三步：写意图分类节点

**练习：** 创建 `src/agent/mainGraph.ts`，先写 `classifyNode`。

这个节点做的事：把用户所有消息拼起来，让 LLM 判断意图。用 `withStructuredOutput` 确保 LLM 返回结构化 JSON。

**答案：**

```typescript
// src/agent/mainGraph.ts
import { ChatOpenAI } from '@langchain/openai'
import { MainState, MainStateType } from './state.js'
import { StateGraph, START, END, interrupt } from '@langchain/langgraph'
import { HumanMessage, AIMessage } from '@langchain/core/messages'
import { z } from 'zod'
import { MemorySaver, Command } from '@langchain/langgraph'

const API_KEY = process.env.QWEN3_API_KEY!
const BASE_URL = 'https://dashscope.aliyuncs.com/compatible-mode/v1'
const MODEL = 'qwen3-max'

const checkpointer = new MemorySaver()

async function classifyNode(state: MainStateType) {
  const model = new ChatOpenAI({
    apiKey: API_KEY,
    configuration: { baseURL: BASE_URL },
    modelName: MODEL,
    temperature: 0,
  }).withStructuredOutput(
    z.object({
      intent: z.enum(['consult', 'apply', 'insufficient']),
      missing: z.array(z.string()),
      documentText: z.string()
    })
  )

  // 拼接所有用户消息
  const allUserText = state.messages
    .filter(m => m instanceof HumanMessage)
    .map(m => m.content)
    .join('\n')

  const reply = await model.invoke([new HumanMessage(`
    分析用户输入，判断意图：
    - consult: 咨询政策信息
    - apply: 要申请加分且信息完整（需要有赛事名称、奖项等级、时间、申请人角色）
    - insufficient: 要申请但缺少信息

    用户输入：
    ${allUserText}
  `)])

  return {
    intent: reply.intent,
    missingInfo: reply.missing || [],
    documentText: reply.documentText || ''
  }
}
```

**`withStructuredOutput` 干了什么？** 它让 LLM 必须返回符合 Zod schema 的 JSON。不用你手动 parse JSON，LangChain 自动处理。如果 LLM 返回的格式不对，会自动重试。

### 第四步：写中断节点

```typescript
// 继续在 mainGraph.ts 里写

async function askForMoreNode(state: MainStateType) {
  const question = `信息不完整，还缺：${state.missingInfo.join('、')}。请补充：`
  const userAnswer = interrupt(question)   // 暂停图执行，等用户补充
  return {
    messages: [
      new AIMessage(question),
      new HumanMessage(String(userAnswer))
    ]
  }
}
```

**`interrupt` 是什么？** 这是 LangGraph 的内置函数。调用它后，**图的执行会暂停**。checkpointer 会把当前状态存下来。后续可以用 `Command({ resume })` 恢复执行，这时 `userAnswer` 就是用户补充的内容。

### 第五步：写咨询子图

**练习：** 创建 `src/agent/consultGraph.ts`，两个节点：
1. `retrieve`：用 MCP 搜索知识库
2. `answer`：把搜索结果 + 用户问题给 LLM，生成回答

**答案：**

```typescript
// src/agent/consultGraph.ts
import { ChatOpenAI } from '@langchain/openai'
import { MainState, MainStateType } from './state.js'
import { HumanMessage, SystemMessage } from '@langchain/core/messages'
import { StateGraph, START, END } from '@langchain/langgraph'
import { callTool } from '../mcp/mcpClient.js'

const API_KEY = process.env.QWEN3_API_KEY!
const BASE_URL = 'https://dashscope.aliyuncs.com/compatible-mode/v1'
const MODEL = 'qwen3-max'

async function retrieveNode(state: MainStateType) {
  const userMsg = state.messages.filter(m => m instanceof HumanMessage).at(-1)!
  const retrievedContext = await callTool('search_knowledge', {
    query: String(userMsg.content), topK: 5
  })
  return { retrievedContext }
}

async function answerNode(state: MainStateType) {
  const userMsg = state.messages.filter(m => m instanceof HumanMessage).at(-1)!
  const model = new ChatOpenAI({
    apiKey: API_KEY,
    configuration: { baseURL: BASE_URL },
    modelName: MODEL,
    temperature: 0.3,
  })
  const reply = await model.invoke([
    new SystemMessage(`你是保研加分助手。根据以下知识库内容回答问题：\n${state.retrievedContext}`),
    new HumanMessage(String(userMsg.content))
  ])
  return { messages: [reply] }
}

export const consultSubgraph = new StateGraph(MainState)
  .addNode('retrieve', retrieveNode)
  .addNode('answer', answerNode)
  .addEdge(START, 'retrieve')
  .addEdge('retrieve', 'answer')
  .addEdge('answer', END)
  .compile()
```

**这就是 RAG 模式：** retrieve（检索）→ answer（生成）。LLM 不是凭空回答，而是基于从知识库搜到的内容来回答。

### 第六步：写申报子图

**练习：** 创建 `src/agent/applyGraph.ts`，三个节点：
1. `fetchPolicy`：用文档内容搜索相关政策
2. `analyzeAndMatch`：LLM 把证书和加分模板做匹配
3. `summarize`：把匹配结果格式化成用户看得懂的回答

**答案：**

```typescript
// src/agent/applyGraph.ts
import { ChatOpenAI } from '@langchain/openai'
import { MainState, MainStateType } from './state.js'
import { StateGraph, START, END } from '@langchain/langgraph'
import { HumanMessage, SystemMessage, AIMessage } from '@langchain/core/messages'
import { z } from 'zod'
import { callTool } from '../mcp/mcpClient.js'

const API_KEY = process.env.QWEN3_API_KEY!
const BASE_URL = 'https://dashscope.aliyuncs.com/compatible-mode/v1'
const MODEL = 'qwen3-max'

async function fetchPolicyNode(state: MainStateType) {
  const policyContext = await callTool('search_knowledge', {
    query: state.documentText.slice(0, 512), topK: 5
  })
  return { policyContext }
}

async function analyzeAndMatchNode(state: MainStateType) {
  if (!state.templates || state.templates.length === 0) {
    return { checkResults: ['{"error":"无可用加分模板"}'] }
  }

  const model = new ChatOpenAI({
    apiKey: API_KEY,
    configuration: { baseURL: BASE_URL },
    modelName: MODEL,
    temperature: 0.1,
  }).withStructuredOutput(z.object({
    suggestions: z.array(z.object({
      templateId: z.number(),
      templateName: z.string(),
      ruleId: z.number(),
      ruleName: z.string(),
      estimatedScore: z.number(),
      reason: z.string(),
    }))
  }))

  const result = await model.invoke([
    new SystemMessage('你是加分审核专家。分析证明材料，判断可以申请哪些加分项。'),
    new HumanMessage(`证明材料：\n${state.documentText.slice(0, 2000)}\n\n加分模板：\n${JSON.stringify(state.templates)}\n\n政策参考：\n${state.policyContext}`)
  ])

  return { checkResults: result.suggestions.map(s => JSON.stringify(s)) }
}

async function summarizeNode(state: MainStateType) {
  const suggestions = state.checkResults
    .map(r => { try { return JSON.parse(r) } catch { return null } })
    .filter(Boolean)

  if (suggestions.length === 0 || suggestions[0]?.error) {
    return { messages: [new AIMessage('暂未匹配到加分项，请确认材料是否完整。')] }
  }

  const summary = suggestions.map((s: any) =>
    `• **${s.templateName}** / ${s.ruleName}：预计 ${s.estimatedScore} 分（${s.reason}）`
  ).join('\n')

  return { messages: [new AIMessage(`匹配到以下加分项：\n\n${summary}`)] }
}

export const applySubgraph = new StateGraph(MainState)
  .addNode('fetchPolicy', fetchPolicyNode)
  .addNode('analyzeAndMatch', analyzeAndMatchNode)
  .addNode('summarize', summarizeNode)
  .addEdge(START, 'fetchPolicy')
  .addEdge('fetchPolicy', 'analyzeAndMatch')
  .addEdge('analyzeAndMatch', 'summarize')
  .addEdge('summarize', END)
  .compile()
```

### 第七步：组装主图

回到 `mainGraph.ts`，把所有节点串起来：

```typescript
// 继续在 mainGraph.ts 里写

import { consultSubgraph } from './consultGraph.js'
import { applySubgraph } from './applyGraph.js'

const mainGraph = new StateGraph(MainState)
  .addNode('classify', classifyNode)
  .addNode('ask', askForMoreNode)
  .addNode('applyGraph', applySubgraph)
  .addNode('consultGraph', consultSubgraph)
  .addEdge(START, 'classify')
  .addConditionalEdges('classify', (s) => s.intent, {
    insufficient: 'ask',
    apply: 'applyGraph',
    consult: 'consultGraph'
  })
  .addEdge('ask', 'classify')       // 补充信息后重新分类
  .addEdge('applyGraph', END)
  .addEdge('consultGraph', END)

const app = mainGraph.compile({ checkpointer })
```

**看这个图结构：**

```
START → classify
          ├── 'consult' → consultGraph → END
          ├── 'apply' → applyGraph → END
          └── 'insufficient' → ask → classify（循环，重新分类）
```

`addConditionalEdges` 就是根据 state.intent 的值决定走哪条边。`checkpointer` 是 `MemorySaver`，把图的状态存在内存里，这样中断后能恢复。

### 第八步：导出调用函数

```typescript
// mainGraph.ts 最后加上

export async function invokeAgent(userInput: string, sessionId: string, documentText = '', templates: any[] = []) {
  const config = { configurable: { thread_id: sessionId } }
  const result = await app.invoke({
    messages: [new HumanMessage(userInput)],
    documentText,
    templates,
  }, config)

  // 检查是否有 interrupt
  const snapshot = await app.getState(config)
  const interrupts = (snapshot.tasks ?? []).flatMap((t: any) => t.interrupts ?? [])
  if (interrupts.length > 0) {
    return { interrupted: true, question: interrupts[0].value }
  }

  const lastAI = result.messages.filter((m: any) => m._getType?.() === 'ai').at(-1)
  return { interrupted: false, reply: String(lastAI?.content ?? '') }
}

export async function resumeAgent(sessionId: string, supplement: string) {
  const config = { configurable: { thread_id: sessionId } }
  const result = await app.invoke(new Command({ resume: supplement }), config)

  const snapshot = await app.getState(config)
  const interrupts = (snapshot.tasks ?? []).flatMap((t: any) => t.interrupts ?? [])
  if (interrupts.length > 0) {
    return { interrupted: true, question: interrupts[0].value }
  }

  const lastAI = result.messages.filter((m: any) => m._getType?.() === 'ai').at(-1)
  return { interrupted: false, reply: String(lastAI?.content ?? '') }
}
```

**`thread_id` 的作用：** 每个会话有自己的 thread_id，checkpointer 用它来区分不同用户的图状态。这样用户 A 的中断不会影响用户 B。

### 今天你搞清楚了什么

- LangGraph 的核心三件套：State（数据）+ Node（处理函数）+ Edge（连接）
- 条件边：根据 state 的值走不同路径
- interrupt / resume：暂停图执行，等用户补充信息后继续
- 子图：consultGraph 和 applyGraph 各自是独立的图，被主图当作节点使用
- checkpointer：存储图状态，使中断/恢复成为可能

## Day 5：加上流式输出（SSE）

今天要让 Agent 的回答一个字一个字地"打"出来，而不是等全部生成完才返回。

### SSE 是什么

Server-Sent Events，HTTP 协议上的服务端推送。服务端不关闭连接，持续发送 `data: ...\n\n` 格式的数据。

### 第一步：给 Agent 加流式输出

**练习：** 在 `mainGraph.ts` 里写一个 `streamAgent` 生成器函数，用 `app.streamEvents` 逐 token 输出。

关键问题：classify 节点也会产生 token（它在生成 JSON），但这些 token 不应该发给用户。需要根据 `event.metadata.langgraph_node` 过滤。

**答案：**

```typescript
// mainGraph.ts 里加上

export async function* streamAgent(
  userInput: string, sessionId: string, documentText = '', templates: any[] = []
) {
  const config = { configurable: { thread_id: sessionId } }
  const SKIP_NODES = new Set(['classify', 'analyzeAndMatch'])

  const eventStream = app.streamEvents(
    { messages: [new HumanMessage(userInput)], documentText, templates },
    { ...config, version: 'v2' }
  )

  for await (const event of eventStream) {
    if (event.event === 'on_chat_model_stream') {
      const node = event.metadata?.langgraph_node
      if (node && SKIP_NODES.has(node)) continue   // 跳过分类和分析节点的 token
      const token = event.data?.chunk?.content
      if (token) yield { type: 'token', data: { content: token } }
    }
  }

  // 流结束后检查是否有 interrupt
  const snapshot = await app.getState(config)
  const interrupts = (snapshot.tasks ?? []).flatMap((t: any) => t.interrupts ?? [])
  if (interrupts.length > 0) {
    yield { type: 'interrupt', data: { question: interrupts[0].value } }
    return
  }
  yield { type: 'result', data: { reply: '完成' } }
}
```

**为什么要跳过 classify 和 analyzeAndMatch？** 因为这两个节点用了 `withStructuredOutput`，LLM 在生成类似 `{"intent":"consult","missing":[]` 这样的 JSON 文本。如果不过滤，用户会看到一堆 JSON 碎片出现在回答前面。只有 answer 和 summarize 节点产生的 token 才是用户要看的回答。

### 第二步：写 SSE 路由

**练习：** 在 Express 里写一个 `POST /agent/stream` 路由，把 streamAgent 的输出转成 SSE 格式。

**答案：**

```typescript
// src/routes/agent.ts（新建或在 server.ts 里加）
import { streamAgent } from '../agent/mainGraph.js'

app.post('/agent/stream', upload.single('file'), async (req, res) => {
  const { message, sessionId } = req.body

  // 设置 SSE 响应头
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  })

  try {
    for await (const event of streamAgent(message, sessionId)) {
      res.write(`data: ${JSON.stringify(event)}\n\n`)
    }
  } catch (err) {
    res.write(`data: ${JSON.stringify({ type: 'error', data: { message: String(err) } })}\n\n`)
  }

  res.write('data: [DONE]\n\n')
  res.end()
})
```

**SSE 的格式规范：** 每条消息是 `data: 内容\n\n`（注意两个换行）。`[DONE]` 是约定的结束标记。

```typescript
async function main() {

}
```

## Day 6：写 Java 代理层（Agent 相关部分）

今天不写 Node 了，写 Java。目标是让 Spring Boot 后端作为中间层，把前端请求转发给 Agent 服务。

### 为什么要这一层

前端不直接调 Agent 的 3001 端口，原因：
1. 后端有权限校验（JWT），Agent 没有
2. 后端要从 MySQL 读加分模板注入到请求里
3. 统一入口，Agent 服务不暴露

### 你要做什么

写两个类：`AIAgentService`（HTTP 客户端）和 `AIAgentController`（控制器）。

### 第一步：AIAgentService — 转发 HTTP 请求

**练习：** 用 OkHttp 写一个 Service，封装对 Agent 的调用。最关键的是 `agentStream` 方法——从 Agent 读 SSE 流，逐行转发给 Spring 的 `SseEmitter`。

**答案：**

```java
@Service
public class AIAgentService {
    @Value("${ai-agent.base-url:http://localhost:3001}")
    private String baseUrl;

    private final OkHttpClient http = new OkHttpClient.Builder()
            .connectTimeout(100, TimeUnit.SECONDS)
            .readTimeout(120, TimeUnit.SECONDS)
            .writeTimeout(300, TimeUnit.SECONDS)
            .build();

    // 流式转发的核心方法
    public void agentStream(String message, String sessionId,
                            MultipartFile file, String templates,
                            SseEmitter emitter) {
        try {
            // 1. 构建 multipart 请求（跟 Agent 的 /agent/stream 接口格式一致）
            MultipartBody.Builder builder = new MultipartBody.Builder()
                    .setType(MultipartBody.FORM)
                    .addFormDataPart("message", message)
                    .addFormDataPart("sessionId", sessionId);
            if (templates != null) builder.addFormDataPart("templates", templates);
            if (file != null && !file.isEmpty()) {
                builder.addFormDataPart("file", file.getOriginalFilename(),
                    RequestBody.create(file.getBytes(), MediaType.parse("application/octet-stream")));
            }

            Request req = new Request.Builder()
                    .url(baseUrl + "/agent/stream")
                    .post(builder.build())
                    .build();

            // 2. 同步调用 Agent，拿到响应流
            try (Response resp = http.newCall(req).execute()) {
                if (resp.body() == null) {
                    emitter.completeWithError(new RuntimeException("Agent 不可达"));
                    return;
                }
                // 3. 逐行读取 Agent 的 SSE 输出
                BufferedReader reader = new BufferedReader(
                    new InputStreamReader(resp.body().byteStream(), StandardCharsets.UTF_8));
                String line;
                while ((line = reader.readLine()) != null) {
                    if (!line.startsWith("data: ")) continue;  // 只处理 data: 开头的行
                    String data = line.substring(6).trim();
                    if ("[DONE]".equals(data)) break;
                    // 4. 转发给前端
                    emitter.send(SseEmitter.event().data(data));
                }
            }
            emitter.complete();
        } catch (Exception e) {
            try { emitter.completeWithError(e); } catch (Exception ignored) {}
        }
    }
}
```

**SSE 转发的数据流：** Agent(Node) → `data: {"type":"token",...}\n\n` → OkHttp 读取 → `reader.readLine()` → 去掉 `data: ` 前缀 → `emitter.send()` → Spring 重新包装成 SSE → 前端。

### 第二步：AIAgentController

**练习：** 写控制器，关键在 `/stream` 接口：创建 `SseEmitter`，异步调用 Service。

**答案：**

```java
@RestController
@RequestMapping("/api/agent")
public class AIAgentController {
    private final AIAgentService aiAgentService;
    private final TemplateMapper templateMapper;  // 从 MySQL 读加分模板

    // 流式接口
    @PostMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter stream(@RequestParam("message") String message,
                             @RequestParam(value = "sessionId", required = false) String sessionId,
                             @RequestParam(value = "file", required = false) MultipartFile file,
                             HttpServletRequest httpRequest) {

        SseEmitter emitter = new SseEmitter(180_000L);  // 3 分钟超时

        // 自动生成 sessionId
        if (sessionId == null || sessionId.isEmpty()) {
            Integer userId = (Integer) httpRequest.getAttribute("userId");
            sessionId = "agent_" + (userId != null ? userId : "anon") + "_" + System.currentTimeMillis();
        }

        // 先发 session 事件给前端（让前端知道 sessionId）
        String sid = sessionId;
        try {
            emitter.send(SseEmitter.event().data(
                "{\"type\":\"session\",\"data\":{\"sessionId\":\"" + sid + "\"}}"));
        } catch (Exception e) {
            emitter.completeWithError(e);
            return emitter;
        }

        // 从 MySQL 查加分模板，序列化成 JSON 传给 Agent
        String templates = loadTemplatesJson();

        // 异步转发（不阻塞 HTTP 线程）
        CompletableFuture.runAsync(() ->
            aiAgentService.agentStream(message, sid, file, templates, emitter)
        );

        return emitter;
    }

    // resume-stream 接口也是类似的模式
    @PostMapping(value = "/resume-stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter resumeStream(@RequestBody Map<String, String> body) {
        SseEmitter emitter = new SseEmitter(180_000L);
        CompletableFuture.runAsync(() ->
            aiAgentService.agentResumeStream(body.get("sessionId"), body.get("supplement"), emitter)
        );
        return emitter;
    }
}
```

**为什么用 `CompletableFuture.runAsync`？** `SseEmitter` 的工作方式是：Controller 返回 emitter 对象后，Spring 保持 HTTP 连接不关闭。然后在另一个线程里通过 `emitter.send()` 持续推送数据。如果不用异步，HTTP 线程会被阻塞住直到流结束。

**`loadTemplatesJson` 干了什么？** 从 MySQL 查所有激活的加分模板和规则，序列化成 JSON。Agent 收到后可以用来做证书-模板匹配。这就是"后端注入数据"的含义——前端不用传模板，后端自己查。

### 今天你搞清楚了什么

- Java 后端的角色：权限校验 + 数据注入 + 请求转发
- SSE 转发的核心：OkHttp 读 Agent 的流 → SseEmitter 推给前端
- `SseEmitter` + `CompletableFuture` 的异步模式
- Controller 额外做的事：生成 sessionId、注入 templates、先发 session 事件

## Day 7：写前端 SSE 消费（Agent 相关部分）

### 第一步：写 SSE 消费函数

**练习：** 写 `consumeSSE` 函数和 `agentStreamChat` 函数。

不用 `EventSource`（只支持 GET，不能传 Header），用 `fetch + ReadableStream`。

**答案：**

```typescript
// api/apiAIagent.ts

const BASE = import.meta.env.VITE_BASE_API

interface AgentCallbacks {
  onToken?: (content: string) => void
  onInterrupt?: (question: string) => void
  onResult?: (data: any) => void
  onSession?: (sessionId: string) => void
  onError?: (msg: string) => void
  onDone?: () => void
}

async function consumeSSE(response: Response, cb?: AgentCallbacks) {
  if (!response.ok || !response.body) {
    cb?.onError?.(`请求失败: ${response.status}`)
    cb?.onDone?.()
    return
  }

  const reader = response.body.getReader()
  const decoder = new TextDecoder()
  let buffer = ''

  while (true) {
    const { done, value } = await reader.read()
    if (done) break

    buffer += decoder.decode(value, { stream: true })
    const lines = buffer.split('\n')
    buffer = lines.pop() || ''    // 最后一行可能不完整，留到下一轮

    for (const line of lines) {
      const trimmed = line.trim()
      if (!trimmed.startsWith('data:')) continue
      const payload = trimmed.replace(/^data:\s*/, '').trim()
      if (payload === '[DONE]') { cb?.onDone?.(); return }

      try {
        const event = JSON.parse(payload)
        switch (event.type) {
          case 'token':     cb?.onToken?.(event.data.content); break
          case 'interrupt': cb?.onInterrupt?.(event.data.question); break
          case 'result':    cb?.onResult?.(event.data); break
          case 'session':   cb?.onSession?.(event.data.sessionId); break
          case 'error':     cb?.onError?.(event.data.message); break
        }
      } catch {}
    }
  }
  cb?.onDone?.()
}

// 发起流式对话
export function agentStreamChat(
  message: string, sessionId: string, file?: File, cb?: AgentCallbacks
): AbortController {
  const controller = new AbortController()
  const token = localStorage.getItem('accessToken') || ''

  const formData = new FormData()
  formData.append('message', message)
  formData.append('sessionId', sessionId)
  if (file) formData.append('file', file)

  fetch(`${BASE}/api/agent/stream`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}` },   // 不设 Content-Type，让浏览器自己加 multipart boundary
    body: formData,
    signal: controller.signal,
  })
    .then(resp => consumeSSE(resp, cb))
    .catch(err => {
      if (err.name !== 'AbortError') cb?.onError?.(String(err))
      cb?.onDone?.()
    })

  return controller   // 外面可以 controller.abort() 取消请求
}

// 中断恢复
export function agentResumeStream(
  sessionId: string, supplement: string, cb?: AgentCallbacks
): AbortController {
  const controller = new AbortController()
  const token = localStorage.getItem('accessToken') || ''

  fetch(`${BASE}/api/agent/resume-stream`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ sessionId, supplement }),
    signal: controller.signal,
  })
    .then(resp => consumeSSE(resp, cb))
    .catch(err => {
      if (err.name !== 'AbortError') cb?.onError?.(String(err))
      cb?.onDone?.()
    })

  return controller
}
```

**buffer 拼接逻辑：** 网络传输可能把一条 SSE 数据拆成多个 chunk。比如 `data: {"type":"tok` 和 `en","data":{"content":"你"}}\n` 会分两次到达。用 buffer 拼接后按 `\n` 切分，就能得到完整的行。最后一个元素 `lines.pop()` 可能是不完整的半行，留到下次拼。

```typescript
// ai-agent/index.vue 里的核心逻辑

const messages = ref<{role: string, content: string}[]>([])
const isLoading = ref(false)
const isInterrupted = ref(false)
const currentSessionId = ref('sess_' + Date.now())

function handleSend() {
  const text = inputMessage.value.trim()
  const file = pendingFile.value

  // 1. 加一条用户消息
  messages.value.push({ role: 'user', content: text || '(上传文件)' })
  // 2. 加一条空的 AI 消息（等 token 填充）
  messages.value.push({ role: 'assistant', content: '' })
  isLoading.value = true

  // 3. 调流式接口
  agentStreamChat(text, currentSessionId.value, file, {
    onToken(content) {
      // 把 token 追加到最后一条 AI 消息
      messages.value.at(-1)!.content += content
    },
    onInterrupt(question) {
      // 显示中断 UI
      messages.value.push({ role: 'interrupt', content: question })
      isInterrupted.value = true
    },
    onSession(id) {
      currentSessionId.value = id
    },
    onDone() {
      isLoading.value = false
    }
  })

  inputMessage.value = ''
  pendingFile.value = null
}

function handleResume() {
  const supplement = supplementInput.value.trim()
  messages.value.push({ role: 'user', content: supplement })
  messages.value.push({ role: 'assistant', content: '' })
  isInterrupted.value = false
  isLoading.value = true

  agentResumeStream(currentSessionId.value, supplement, {
    onToken(content) {
      messages.value.at(-1)!.content += content
    },
    onInterrupt(question) {
      messages.value.push({ role: 'interrupt', content: question })
      isInterrupted.value = true
    },
    onDone() {
      isLoading.value = false
    }
  })
}
```

**流式显示的原理：** 先添加一条空的 assistant 消息，然后每收到一个 token 就追加到这条消息的 content 里。Vue 的响应式系统会自动更新 DOM，用户就看到文字一个个蹦出来。

**中断/恢复的 UI 流程：**
1. 收到 `interrupt` 事件 → `isInterrupted = true` → 显示一个橙色气泡和补充信息输入框
2. 用户输入补充信息 → `handleResume()` → 调 `agentResumeStream` → Agent 从中断处继续执行
3. 继续接收 token → 显示最终回答

- 流式显示就是"空消息 + 逐 token 追加"
- 中断 UI 就是一个状态标志 `isInterrupted` 控制显示
