## Day 2：文档解析与向量化（核心难点）

### 学习目标
完全理解"文档 → 文本 → 分块 → 向量 → 存储"的全流程。

### 学习内容

#### 2.1 文档解析器 (`src/services/docParser.ts`)

```
文件：ID-AIDemo/src/services/docParser.ts
关键函数：parseFile(filePath, ext) 和 chunkText(text, chunkSize, overlap)
```

**parseFile 支持的格式：**

| 格式 | 处理方式 |
|------|---------|
| PDF | `pdf-parse` 库提取全部页面文本 |
| DOCX/DOC | `mammoth` 库提取原始文本 |
| XLSX/XLS | `xlsx` 库逐行逐列拼接为制表符分隔文本 |
| MD/TXT | 直接 `fs.readFileSync` 读取 UTF-8 |

**chunkText 分块策略：**

```typescript
chunkText(text, chunkSize = 500, overlap = 100)
```

- 滑动窗口法：每次取 500 字符，步进 400（500 - 100 overlap）
- 过滤掉长度 < 30 的块（太短没有语义信息）
- overlap 确保上下文不会在块边界处断裂

**思考：** 为什么要分块？因为 embedding 模型对输入长度有限制（这里截断到 512 字符），且短文本的向量更精确地表达局部语义。

#### 2.2 Embedding 向量化 (`src/services/embeddings.ts`)

```
文件：ID-AIDemo/src/services/embeddings.ts
关键函数：getEmbedding(text) 和 getEmbeddings(texts)
```

**核心流程：**

```
文本 → 截断到512字符 → POST /embeddings API → 返回 number[]
```

- 调用的是 OpenAI 兼容接口（DashScope），模型默认 `text-embedding-v3`
- `getEmbedding`：单条文本 → 单个向量
- `getEmbeddings`：批量文本 → 批量向量，**每批最多 6 条**（避免超时）
- 每次调用从 `aiConfig` 获取最新的 `api_key`、`base_url`、`embedding_model`

**关键代码逻辑：**

```typescript
// 单条 embedding
async function getEmbedding(text: string): Promise<number[]> {
    const truncated = text.slice(0, 512);
    const response = await fetch(`${BASE_URL}/embeddings`, {
        method: 'POST',
        headers: { 'Authorization': `Bearer ${API_KEY}`, 'Content-Type': 'application/json' },
        body: JSON.stringify({ model: EMBEDDING_MODEL, input: truncated })
    });
    return response.data[0].embedding; // number[] 例如 [0.012, -0.034, ...]
}
```

**向量的本质：** 一个 1024 维（取决于模型）的浮点数组，语义相似的文本在向量空间中距离更近。

#### 2.3 向量存储 (`src/services/vectorStore.ts`)

```
文件：ID-AIDemo/src/services/vectorStore.ts
关键函数：saveChunk, searchSimilar, deleteBySourceFile
```

**存储方式：**

```sql
INSERT INTO knowledge_chunks (source_file, chunk_index, content, embedding)
VALUES ('政策文件.pdf', 0, '第一章 总则...', '[0.012, -0.034, ...]')
```

- `embedding` 字段存的是 **JSON.stringify(number[])** — 向量序列化为字符串

**搜索方式（暴力搜索 + 余弦相似度）：**

```typescript
function searchSimilar(queryEmbedding: number[], topK: number = 5) {
    const allChunks = db.prepare('SELECT * FROM knowledge_chunks').all();
    // 对每个 chunk 计算与 queryEmbedding 的余弦相似度
    // 排序取 topK
    return sorted.slice(0, topK);
}
```

**余弦相似度公式：**

```
cosine(A, B) = (A · B) / (|A| × |B|)
```

值域 [-1, 1]，1 表示完全相同方向，0 表示正交（不相关），-1 表示完全相反。

**为什么用 SQLite + 暴力搜索而不用向量数据库？**
- 数据量小（几十到几百个 chunk），暴力搜索够用
- 部署简单，无需额外服务（Milvus/Pinecone/pgvector 等都需要独立部署）
- 如果未来数据增长，可以迁移到专业向量数据库

#### 2.4 两条数据导入路径

**路径 A：启动时自动导入（seedKnowledge）**

```
文件：ID-AIDemo/src/seed/seedKnowledge.ts
```

```
启动 → 扫描 docs/加分文件/ → 跳过已导入的文件
     → parseFile → chunkText → getEmbeddings → saveChunk（直接调用，不经 MCP）
```

```

```


两条路径最终都写入 **同一张 `knowledge_chunks` 表**。

## Day 3：MCP（Model Context Protocol）机制

### 学习目标
理解 MCP 是什么、为什么需要 MCP、以及在本项目中的具体实现。

### 学习内容

#### 3.1 MCP 的作用

MCP 是 Anthropic 提出的标准协议，让 LLM 能调用外部工具。在本项目中：

```
LangGraph Agent ──callTool──▶ MCP Client ──stdio──▶ MCP Server (knowledgeServer)
                                                         │
                                                    操作 SQLite
                                                    调用 Embedding API
```

**为什么不直接调用 vectorStore？**
- MCP 将知识库操作封装为**标准化工具**，Agent 图中的节点只需 `callTool('search_knowledge', {query})` 而不需要知道底层实现
- MCP Server 运行在**独立子进程**中，与 Agent 主进程隔离
- 未来可以替换为远程 MCP Server 而不改 Agent 代码

#### 3.2 MCP Server (`src/mcp/knowledgeServer.ts`)

```
文件：ID-AIDemo/src/mcp/knowledgeServer.ts
```

这是一个 **stdio 类型的 MCP Server**，通过标准输入输出与 Client 通信。

**注册的 5 个工具：**

| 工具名 | 输入 | 功能 |
|--------|------|------|
| `search_knowledge` | query: string, topK: number | 向量搜索知识库，返回最相似的 chunk |
| `parse_document` | filePath: string, ext: string | 解析文档为纯文本 |
| `ingest_document` | filePath: string, ext: string, sourceFile: string | 解析 + 分块 + 向量化 + 存储（一条龙） |
| `list_knowledge_sources` | 无 | 列出已导入的文件 |
| `delete_knowledge_source` | sourceFile: string | 删除指定文件的所有 chunk |

**`search_knowledge` 内部流程：**

```
query → getEmbedding(query) → searchSimilar(embedding, topK)
      → 格式化为文本返回给 Agent
```

#### 3.3 MCP Client (`src/mcp/mcpClient.ts`)

```
文件：ID-AIDemo/src/mcp/mcpClient.ts
```

- 使用 `@langchain/mcp-adapters` 的 `MultiServerMCPClient`
- **懒加载**：第一次调用时才启动 MCP Server 子进程
- 传输方式：**stdio**，即通过子进程的 stdin/stdout 通信
- 启动命令：`node --import tsx src/mcp/knowledgeServer.ts`

**关键函数：**

```typescript
// 获取所有工具的 LangChain 包装
async function getKnowledgeTools(): Promise<Tool[]>

// 按名称调用工具
async function callTool(name: string, input: object): Promise<string>

// 关闭 MCP 连接
async function closeMcpClient(): Promise<void>
```

#### 3.4 MCP 在 Agent 图中的调用位置

- **consultGraph** 的 `retrieve` 节点：`callTool('search_knowledge', { query, topK: 5 })`
- **applyGraph** 的 `fetchPolicy` 节点：`callTool('search_knowledge', { query: documentText前512字符 })`
- **knowledge 路由**的上传/删除：`callTool('ingest_document', ...)` / `callTool('delete_knowledge_source', ...)`

## Day 4：LangGraph Agent 核心（主图 + 子图）

### 学习目标
完全理解 Agent 的图结构、节点逻辑、状态流转和中断/恢复机制。

### 学习内容

#### 4.1 State 定义 (`src/agent/state.ts`)

```
文件：ID-AIDemo/src/agent/state.ts
```

LangGraph 的核心概念是 **有状态图（StateGraph）**。所有节点共享同一个 State：

```typescript
MainState = {
    messages: BaseMessage[],       // 聊天历史（LangGraph 内置 reducer，自动追加）
    intent: string,                 // 分类结果：'consult' | 'apply' | 'insufficient'
    missingInfo: string[],          // 申报流程中缺少的信息
    documentText: string,           // 上传文档的解析文本
    checkResults: string[],         // 申报匹配结果（append reducer）
    retrievedContext: string,       // 检索到的知识库内容
    answerDraft: string,            // 回答草稿
    templates: ScoreTemplate[],     // 加分模板
    policyContext: string           // 政策上下文
}
```

#### 4.2 主图 (`src/agent/mainGraph.ts`)

```
文件：ID-AIDemo/src/agent/mainGraph.ts
```

**图结构：**

```
START
  │
  ▼
classify（意图分类）
  │
  ├── intent='consult'      → consultGraph（咨询子图）→ END
  ├── intent='apply'        → applyGraph（申报子图）  → END
  └── intent='insufficient' → ask（中断，要求补充信息）→ classify（重新分类）
```

**classify 节点：**
- 使用 `ChatOpenAI` + `withStructuredOutput`（Zod schema）
- LLM 被要求输出结构化 JSON：`{ intent, missing, documentText }`
- 不是简单的关键词匹配，而是 LLM 理解语义后分类

**ask 节点（中断机制）：**

```typescript
function askForMoreNode(state) {
    const question = `请补充以下信息：${state.missingInfo.join('、')}`;
    return { messages: [interrupt(question)] };
}
```

- `interrupt()` 是 LangGraph 的内置函数，会**暂停图执行**
- 前端收到中断信号后显示补充信息输入框
- 用户提交后通过 `resumeAgent` 继续执行

**MemorySaver：**
- `const checkpointer = new MemorySaver()` — 内存中的检查点
- `thread_id = sessionId`：每个会话独立的执行状态
- 使得中断后可以恢复执行（因为图状态被保存了）

#### 4.3 咨询子图 (`src/agent/subGraphs/consultGraph.ts`)

```
文件：ID-AIDemo/src/agent/subGraphs/consultGraph.ts
```

**流程：**

```
retrieve → answer → END
```

- **retrieve 节点**：调用 MCP `search_knowledge` 获取相关知识
- **answer 节点**：将检索到的知识 + 用户问题一起送给 LLM 回答

这就是经典的 **RAG（检索增强生成）** 模式：先检索，再回答。

#### 4.4 申报子图 (`src/agent/subGraphs/applyGraph.ts`)

```
文件：ID-AIDemo/src/agent/subGraphs/applyGraph.ts
```

**流程：**

```
fetchPolicy → analyzeAndMatch → summarize → END
```

- **fetchPolicy**：用文档前 512 字符搜索相关政策
- **analyzeAndMatch**：LLM 结构化输出，将证书与模板规则匹配
- **summarize**：将匹配结果整理为用户友好的 Markdown 回复

#### 4.5 流式输出与事件过滤

```typescript
// streamAgent 中的关键逻辑
const SKIP_NODES = ['classify', 'analyzeAndMatch'];

for await (const event of stream) {
    if (event.event === 'on_chat_model_stream') {
        const node = event.metadata?.langgraph_node;
        if (SKIP_NODES.includes(node)) continue; // 跳过非回答节点的 token
        yield { type: 'token', content: event.data.chunk.content };
    }
}
```

为什么要跳过？因为 `classify` 节点输出的是结构化 JSON（意图分类），不应该显示给用户。

## Day 5：AI 配置与运行时热更新

### 学习目标
理解 AI 配置如何存储、读取、缓存，以及管理员修改配置后如何生效。

### 学习内容

#### 5.1 配置服务 (`src/services/aiConfig.ts`)

```
文件：ID-AIDemo/src/services/aiConfig.ts
```

**配置项：**

| Key | 默认值 | 含义 |
|-----|--------|------|
| system_role | 保研助手提示词 | LLM 的系统角色设定 |
| api_key | 环境变量 `QWEN3_API_KEY` | 模型 API 密钥 |
| base_url | 环境变量 `QWEN_BASE_URL` | API 基础 URL |
| chat_model | `qwen3-max` | 对话模型 |
| embedding_model | `text-embedding-v3` | 向量化模型 |

**缓存机制：**

```typescript
let cache = null;
let cacheTime = 0;
const CACHE_TTL = 60_000; // 60 秒

function loadAll() {
    if (cache && Date.now() - cacheTime < CACHE_TTL) return cache;
    cache = db.prepare('SELECT * FROM ai_config').all();
    cacheTime = Date.now();
    return cache;
}
```

- 内存缓存 60 秒，避免每次调用都查数据库
- `updateConfig` 会清除缓存（`invalidate()`）

**特别注意 MCP 子进程的缓存问题：**
MCP Server 运行在独立子进程中，有自己的 `aiConfig` 缓存。管理员更新配置后：
- 主进程的缓存立即失效 ✅
- MCP 子进程的缓存需要等 TTL 到期才更新（最多 60 秒延迟）⚠️

#### 5.2 完整的配置更新链路

```
管理员在 Admin UI 修改配置
  │
  ▼
PUT /api/ai-config (idfrontend-admin → idbackend)
  │
  ▼
AIConfigController → AIAgentService.updateAIConfig()
  │
  ▼
PUT /config (idbackend → ID-AIDemo)
  │
  ▼
routes/config.ts → aiConfig.updateConfig() → SQLite UPDATE + 缓存失效
  │
  ▼

```

#### 6.3 SSE 转发机制

这是理解流式交互的关键。以 `agentStream` 为例：

```
前端 fetch → Java SseEmitter → OkHttp → Node Express SSE
                 ↑                           │
                 └───── 逐行转发 ◄────────────┘
```

**Java 端核心逻辑：**

```java
// Controller
@PostMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter agentStream(...) {
    SseEmitter emitter = new SseEmitter(180_000L);  // 3 分钟超时
    CompletableFuture.runAsync(() -> {
        aiAgentService.agentStream(message, sessionId, file, templates, emitter);
    });
    return emitter;
}

// Service
void agentStream(..., SseEmitter emitter) {
    // 1. 构建 multipart 请求
    // 2. OkHttp 同步调用 Node /agent/stream
    // 3. BufferedReader 逐行读取响应
    while ((line = reader.readLine()) != null) {
        if (line.startsWith("data: ")) {
            String data = line.substring(6).trim();
            if (data.equals("[DONE]")) break;
            emitter.send(SseEmitter.event().data(data));  // 转发给前端
        }
    }
    emitter.complete();
}
```

**注意事项：**
- `SseEmitter` 是 Spring 的 SSE 支持，Controller 直接返回它
- 实际的 HTTP 读取在异步线程中进行（`CompletableFuture.runAsync`）
- 逐行转发：Java 读到 Node 的 `data: {...}` 后，去掉 `data: ` 前缀，再通过 `emitter.send` 发给前端

#### 6.4 AIAgentController 的额外职责

```
文件：idbackend/.../controller/businessController/AIAgentController.java
```

Controller 不仅仅是转发，还做了：
1. **sessionId 生成**：`agent_{userId}_{timestamp}` 或 `agent_anon_{timestamp}`
2. **模板注入**：`loadTemplatesJson()` 从 MySQL 读取活跃的加分模板和规则
3. **session 事件**：在流式响应开头，先发一个 `{"type":"session","data":{"sessionId":"..."}}` 事件

## Day 7：前端 SSE 消费与 Agent 交互 UI

#### 7.1 API 层 SSE 实现

```
文件：idfrontend/src/api/components/apiAIagent.ts
      idfrontend-admin/src/api/components/apiAIagent.ts
```

**不使用 `EventSource`，而是使用 `fetch` + `ReadableStream`：**

```typescript
async function consumeSSE(response, callbacks) {
    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        buffer += decoder.decode(value, { stream: true });

        // 按换行分割，处理每一行
        const lines = buffer.split('\n');
        buffer = lines.pop(); // 最后一个可能不完整，留到下次

        for (const line of lines) {
            if (line.startsWith('data:')) {
                const data = line.replace(/^data:\s*/, '').trim();
                if (data === '[DONE]') { callbacks.onDone?.(); return; }
                const event = JSON.parse(data);
                // 根据 event.type 分发
                switch (event.type) {
                    case 'token':     callbacks.onToken(event.data.content); break;
                    case 'interrupt': callbacks.onInterrupt(event.data.question); break;
                    case 'result':    callbacks.onResult(event.data); break;
                    case 'session':   callbacks.onSession(event.data.sessionId); break;
                    case 'error':     callbacks.onError(event.data.message); break;
                }
            }
        }
    }
}
```

**为什么用 fetch 而不用 EventSource？**
- `EventSource` 不支持 POST 请求（只能 GET）
- `EventSource` 不支持自定义 Header（需要传 Authorization）
- `fetch` + `ReadableStream` 更灵活

## 附录 A：核心概念速查

### Embedding（向量嵌入）
将文本映射为高维数值向量（如 1024 维），使语义相似的文本在向量空间中距离更近。本工程使用 DashScope 的 `text-embedding-v3` 模型。

### RAG（检索增强生成）
不是让 LLM 从自身知识回答，而是先从知识库检索相关内容，拼接到 prompt 中再让 LLM 回答，保证回答基于特定文档。

### MCP（Model Context Protocol）
Anthropic 提出的标准协议，允许 LLM/Agent 以统一方式调用外部工具。本工程使用 stdio 传输方式（主进程与子进程通过标准输入输出通信）。

### LangGraph StateGraph
有状态的图执行引擎。节点是函数，边可以是条件的。所有节点共享一个 State 对象，通过 reducer 控制更新。支持 checkpointer 保存执行状态，支持 interrupt 暂停和 resume 恢复。

### SSE（Server-Sent Events）
HTTP 协议上的服务端推送机制。服务端持续发送 `data: ...\n\n` 格式的事件，客户端实时接收。本工程用于流式输出 LLM 的回答 token。

### Structured Output（结构化输出）
使用 Zod schema 约束 LLM 输出为 JSON 格式，确保 classify 节点返回的 `{intent, missing, documentText}` 格式正确。
