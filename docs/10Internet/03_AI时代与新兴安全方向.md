# AI 时代与新兴安全方向

## 一、AI 生成代码的安全问题

AI 辅助编程（Copilot、Cursor、ChatGPT）在引入新的安全风险：

- **AI 生成的代码看起来能跑，但不一定安全**。AI 倾向于给"最常见的实现"，而不是"最安全的实现"
- 研究表明 AI 生成的代码中包含已知漏洞模式的比例并不低（不安全的随机数生成、硬编码密钥、不充分的输入校验）
- 开发者对 AI 生成代码的信任度高，review 的仔细程度反而下降 —— **"AI 写的应该没问题吧"这种心态本身就是漏洞**

## 二、LLM 应用本身的漏洞（全新领域）

### 2.1 Prompt Injection（提示词注入）

最经典的 LLM 漏洞，类似于 SQL 注入，但注入的是自然语言指令。

- **直接注入**：用户直接输入恶意 prompt，让 LLM 做不该做的事
- **间接注入**：在 LLM 会读取的外部数据（网页、文档、邮件）里藏入指令，LLM 读到后执行攻击者的意图

### 2.2 Agent 工具调用越权

AI Agent 可以调用工具（搜索、执行代码、访问数据库、发邮件）。如果 Agent 被 Prompt Injection 操控，它会用自己的工具权限执行攻击者想做的事。

> 例：一个能读写数据库的 Agent，被注入后把用户数据导出发到外部。

### 2.3 RAG 数据投毒和泄露

- RAG 系统从知识库检索信息给 LLM 参考
- 如果知识库的数据可以被用户上传影响，攻击者可以注入恶意内容
- 通过精心构造的提问，可以让 LLM 泄露知识库中不该暴露的内容

### 2.4 模型越狱（Jailbreak）

- 绕过 LLM 的安全限制，让它输出不该输出的内容
- 攻防迭代非常快，每次模型更新都有新的绕过方式出现

### 2.5 OWASP LLM Top 10

OWASP 发布的 LLM 应用十大安全风险，是 AI 安全领域的参考标准，值得重点学习。

## 三、Web3 安全方向

### 特殊性

- **智能合约一旦部署就不可修改**，漏洞后果比 Web2 严重得多（钱直接被转走）
- 2024-2025 年因合约漏洞损失的资金仍然在数十亿美元级别

### 常见漏洞类型

| 漏洞 | 说明 |
|------|------|
| **重入攻击** | 合约在更新状态之前调用外部合约，外部合约递归调用原合约 |
| **整数溢出** | Solidity 早期版本没有溢出检查 |
| **权限控制缺陷** | 关键函数没有 onlyOwner 或类似限制 |
| **闪电贷攻击** | 利用无抵押贷款在单笔交易内操纵价格 |
| **预言机操纵** | 操纵价格预言机的数据源 |
| **签名验证绕过** | 签名重放、签名可锻性 |

### 审计工具

- **Slither**：静态分析工具
- **Mythril**：符号执行分析
- **Foundry**：智能合约开发和测试框架

## 四、为什么 AI 安全是一个好的切入点

1. **领域太新了**，还没有成熟的防御标准和最佳实践
2. **OWASP LLM Top 10** 才刚发布不久，行业还在摸索阶段
3. **懂 AI Agent 开发 + 懂安全的人极少**，两边都有基础就是稀缺人才
4. **企业正在大规模部署 LLM 应用**，但安全意识远远没跟上

## 五、差异化方向建议

结合 Web 开发 + AI Agent + Web3 的综合背景：

| 方向 | 具体内容 |
|------|---------|
| **AI 安全** | Prompt Injection 防御、Agent 工具调用安全审计、RAG 数据投毒检测、LLM 应用渗透测试 |
| **Web3 安全** | 智能合约审计、DeFi 协议安全分析、钱包安全 |
| **代码审计** | Java/TypeScript/Python 项目的白盒安全审计，发现业务逻辑漏洞 |
| **安全开发** | 把安全融入 DevOps 流程，开发安全检测工具 |
