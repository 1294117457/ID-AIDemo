# 漏洞类型与攻击面详解

## 一、代码层面的漏洞

### 1.1 注入类漏洞

代码里把用户输入直接拼进去了，本质都是 **把数据当成了代码执行**。

| 漏洞类型 | 原理 | 示例 |
|---------|------|------|
| **SQL 注入** | 用户输入拼进 SQL 语句 | `SELECT * FROM users WHERE name='` + 用户输入 + `'` |
| **XSS** | 用户输入拼进 HTML 页面 | `<div>` + 用户输入 + `</div>`，输入含 `<script>` |
| **命令注入** | 用户输入拼进系统命令 | `os.system("ping " + 用户输入)` |
| **SSTI** | 用户输入拼进模板引擎 | 模板引擎解析用户输入导致代码执行 |

### 1.2 认证与授权类漏洞

权限控制没做好：

- **越权访问**：普通用户能访问管理员接口
  - 水平越权：用户 A 能看到用户 B 的数据
  - 垂直越权：普通用户能执行管理员操作
- **JWT 密钥太弱或没验证签名**：攻击者可以伪造 token
- **密码重置逻辑缺陷**：重置链接没有过期时间、可被预测

### 1.3 业务逻辑类漏洞

代码语法没错，但业务流程有漏洞：

- 支付金额可以改成负数（反向充值）
- 优惠券可以重复使用
- 短信验证码可以爆破（没有频率限制）
- 接口顺序可以跳过（跳过支付直接确认订单）

> 这类漏洞是自动化工具扫不出来的，需要人工理解业务逻辑。

## 二、配置层面的漏洞

代码本身没问题，但部署和配置出了错：

| 问题 | 风险 |
|------|------|
| 数据库暴露在公网，没设密码 | 任何人可以连接并读取全部数据 |
| Debug 模式在生产环境没关 | 泄露堆栈信息、源码路径、环境变量 |
| 默认密码没改（admin/admin） | 攻击者直接登录管理后台 |
| 不必要的端口开放 | 扩大攻击面 |
| .git 目录暴露 | 可以还原全部源代码 |
| 错误信息过于详细 | 泄露数据库类型、表结构等信息 |

## 三、人的漏洞（社会工程学）

不攻击系统，攻击人：

- **钓鱼邮件**：伪造成公司内部邮件，诱导点击链接或下载附件
- **伪装身份**：假装 IT 部门打电话要密码
- **信息泄露**：在 GitHub 上不小心提交了密钥、密码
- **尾随进入**：跟在员工后面进入门禁区域

---

## 四、供应链漏洞

你用的第三方组件有安全问题：

- **第三方库漏洞**：Log4j（CVE-2021-44228）影响了几乎整个 Java 生态
- **npm 包被投毒**：恶意包伪装成热门包名的拼写变体
- **Docker 镜像被篡改**：从非官方源拉取的镜像可能含后门
- **CI/CD 流水线被攻击**：构建过程中注入恶意代码

---

## 五、高级攻击视角

### 5.1 时序攻击 / 侧信道攻击

不直接攻击代码逻辑，而是通过 **观察系统的行为差异** 来推断信息：

- 密码比较时逐字符匹配，第一个字符不匹配就返回失败，响应时间比匹配到第5个字符才失败要短
- 攻击者可以通过响应时间差异逐字符爆破密码
- 非常隐蔽，不触发任何告警

### 5.2 竞态条件（Race Condition）

并发请求之间的时间差导致的漏洞：

- 同时发两个请求取钱，余额只扣了一次
- 同时使用同一张优惠券
- 在权限检查和实际操作之间的微秒时间差里做手脚

### 5.3 反序列化漏洞

把用户可控的数据反序列化时，可能触发任意代码执行。Java 生态里这类漏洞影响巨大（如 Apache Commons Collections 反序列化链）。Spring Boot 项目需要特别关注。

### 5.4 SSRF（服务端请求伪造）

让服务器代替攻击者发起请求，访问内网资源、读取本地文件等。

### 5.5 DNS 重绑定

绕过浏览器同源策略，让浏览器以为在访问外部域名，实际请求被导向内网地址。

## 六、为什么已知漏洞仍然大量存在

已知漏洞类型都有成熟的防护方案，但现实中仍然大量存在：

| 漏洞 | 防护方案 | 现实情况 |
|------|---------|---------|
| SQL 注入 | 参数化查询/ORM | 大量项目仍然有手写拼接 SQL，尤其老项目、内部系统 |
| XSS | 输出编码、CSP | 框架默认防了，但 `v-html`、`dangerouslySetInnerHTML` 一用就破防 |
| CSRF | Token 验证 | 不少项目觉得"用了 JWT 就不需要管 CSRF" |
| 越权 | RBAC 权限校验 | 开发者经常只校验"有没有登录"，没校验"这条数据是不是你的" |
| 配置泄露 | 安全基线扫描 | 上线赶进度，安全配置往往最后做，经常忘 |

**核心矛盾：安全知识和开发知识之间存在巨大的鸿沟。** 大多数开发者不系统学安全，大多数安全工程师不写业务代码。

### 系统复杂度在增长

十年前一个 Web 应用是单体服务。现在一个项目可能涉及：

- 前端 SPA + 后端微服务 + API Gateway
- Redis / MQ / ES / MongoDB 多种中间件
- Docker / K8s 容器化部署
- CDN / OSS / 第三方 OAuth
- CI/CD 流水线

**每多一层架构，就多一层攻击面。系统越复杂，出错的概率越高。**
