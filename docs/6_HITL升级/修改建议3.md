> 基于用户反馈的三个问题，结合代码追踪，给出每个 bug 的精确原因和具体改法。

## Bug 1：开启几轮对话后上传文件按钮消失

### 根因

`score/index.vue` 中，输入区使用了 `v-if / v-else-if` 互斥渲染：

```html
<!-- 当 aiInterrupted = true 时（AI 追问信息不足），只显示文字补充框 -->
<div v-if="aiInterrupted && !aiContextLimitReached">
  <el-input v-model="aiSupplement" ... />
  ...
</div>

<!-- 当 aiInterrupted = false 时才显示含上传按钮的正常输入区 -->
<div v-else-if="!aiContextLimitReached">
  <label ...>  ← 文件上传按钮
  <el-input v-model="aiInput" ... />
  ...
</div>
```

**`askForMoreNode` 触发 interrupt 后，前端 `onInterrupt` 进入 `else` 分支：**
```typescript
// onInterrupt(question, extra) 中，extra.requireFiles 为 false（没有 suggestions）
aiInterrupted.value = true  // ← 设为 true
```

此时文字补充框出现，上传按钮消失。用户在"信息不足"状态下只能输文字，**无法通过上传文件来补充**。

经过几轮追问后，用户发现一直没有上传入口，就是这个原因。

### 修复方案

**合并两个输入区为一个，通过发送逻辑区分操作。**

1. **删除单独的补充输入框**，只保留一个统一输入区（含文件上传按钮）
2. 当 `aiInterrupted = true` 时，在输入框上方显示一条 **amber 色提示横幅**，告知用户 AI 正在等待补充
3. 点击发送时，根据 `aiInterrupted` 决定调用 `agentStreamChat` 还是 `agentResumeStream`
4. `handleAiSend` 统一处理，内部判断：

```typescript
const handleAiSend = () => {
  const msg = aiInput.value.trim()
  const file = aiPendingFile.value
  if ((!msg && !file) || aiLoading.value) return

  aiMessages.value.push({ role: 'user', content: file ? `${msg}\n📎 ${file.name}` : msg })
  aiInput.value = ''
  aiPendingFile.value = null
  const wasInterrupted = aiInterrupted.value   // ← 记录当前状态
  aiInterrupted.value = false
  aiInterruptSuggestions.value = []
  scrollAiToBottom()

  aiMessages.value.push({ role: 'assistant', content: '' })
  const idx = aiMessages.value.length - 1
  aiLoading.value = true
  aiStreaming.value = false

  if (wasInterrupted) {
    // 信息补充 → 走 resume
    aiController = agentResumeStream(aiSessionId.value, msg, buildAiCallbacks(idx))
  } else {
    // 正常发送
    aiController = agentStreamChat(msg, aiSessionId.value, file ?? undefined, buildAiCallbacks(idx))
  }
}
```

5. 删除 `handleAiResume`、`aiSupplement` 变量（不再需要）

**模板改为：**
```html
<div v-if="!aiContextLimitReached" class="mt-3">
  <!-- AI 追问提示横幅 -->
  <div v-if="aiInterrupted" class="text-xs text-amber-600 bg-amber-50 border border-amber-200 rounded-lg px-3 py-1.5 mb-2">
    💬 AI 正在等待您补充信息，可输入文字或上传文件后点击发送
  </div>
  <div class="flex gap-2">
    <label ...>  <!-- 文件上传，始终可见 -->
      <input type="file" ... @change="onAiFileChange" />
    </label>
    <el-input
      v-model="aiInput"
      :placeholder="aiInterrupted ? '请补充信息或上传相关文件...' : '输入消息或上传证书文件...'"
      @keyup.enter="handleAiSend"
      :disabled="aiLoading"
    />
    <el-button type="primary" @click="handleAiSend" :loading="aiLoading" ...>发送</el-button>
  </div>
</div>
```

## Bug 2：Agent 在 resume 后重复上一次回答

### 根因

**LangGraph 恢复中断节点时，会从节点函数头部重新执行整个节点。**

`askForMoreNode`（图中注册名为 `'ask'`，见 `mainGraph.ts` 第 79 行）在我上次修改后变为：

```typescript
async function askForMoreNode(state: MainStateType) {
  const allUserText = ...
  const model = createChatModel(0)
  const reply = await model.invoke(...)     // ← LLM 调用在 interrupt 之前
  const question = String(reply.content)

  const userAnswer = interrupt(question)    // ← 第一次运行到这里才暂停
  return { messages: [...] }
}
```

**第一次运行**：LLM 被调用 → 流式 tokens → `interrupt()` 暂停。
**Resume 时**：LangGraph 重新执行该函数 → LLM 再次被调用 → **又一次流式输出** → 这些 tokens 就是用户看到的"重复的上一次回答"。之后 `interrupt()` 才返回用户的 resume 值，节点完成，图继续运行。

同样地，`applySubgraph` 里的 `summarizeNode`（注册名 `'summarize'`）在 resume 流程中也会先发出一段"为您匹配到以下加分项"的 tokens，再被后续的 `confirmNode` interrupt 覆盖，造成内容闪烁。

### 修复方案

在 `agentService.ts` 中，`streamAgent` 和 `streamResume` 都有一个 `SKIP_NODES` 集合，在 resume 时把被重放的节点加进去即可：

```typescript
// 修改前
const SKIP_NODES = new Set(['classify', 'analyzeAndMatch'])

// 修改后（两处都改）
const SKIP_NODES = new Set(['classify', 'analyzeAndMatch', 'ask', 'summarize'])
```

- `'ask'` — `askForMoreNode` 的节点名，防止 resume 时 LLM 重放 tokens
- `'summarize'` — 防止 summarize 的摘要 tokens 出现在 confirm interrupt 之前，避免内容闪烁

> **注意**：`'ask'` 在 `streamAgent`（首次调用）里也会被跳过，但这没问题——首次的 interrupt 问题是通过 `checkInterrupt` 检测后作为 `interrupt` 事件发给前端的，不依赖 token 流。

## Bug 3：AI 返回申请信息后未显示确认按钮

### 根因

这是一个**子图状态不可见**的问题，是架构层面的 Bug。

`agentService.ts` 中检测 interrupt 后，从主图快照中读取 suggestions：

```typescript
// streamAgent 和 streamResume 里的写法
const snapshot = await app.getState(config)
const state = snapshot.values as any
const suggestions = (state.checkResults ?? [])   // ← 永远是空数组！
  .map(...).filter(Boolean)
```

**`checkResults` 定义在 `ApplyState`（子图），不在 `MainState`（主图）。**

当 `confirmNode` 在子图中触发 `interrupt(question)` 时，子图还没有"返回"（它被中断了），主图的 `snapshot.values` 不包含子图的内部状态。所以 `state.checkResults` 是 `undefined`，`suggestions` 永远是 `[]`，`requireFiles = false`。

前端 `onInterrupt` 进入 `else` 分支（显示文字补充框），而不是推送 `confirm-action` 气泡。

### 修复方案

**让 `confirmNode` 通过 interrupt 值本身携带 suggestions，而不是依赖主图状态。**

#### 改动 1：`applyGraph.ts` — `confirmNode` 传结构化中断值

```typescript
// 修改前
const userAnswer = interrupt(question)

// 修改后：传对象，包含 type 标识 + suggestions
const userAnswer = interrupt({
  type: 'confirm' as const,
  question,
  suggestions,
})
```

这样 `interrupt()` 的 VALUE 本身就携带了 suggestions，不需要从主图状态读取。

#### 改动 2：`agentService.ts` — `checkInterrupt` 识别结构化中断

```typescript
async function checkInterrupt(config: ...) {
  const app = await getApp()
  const snapshot = await app.getState(config)
  const interrupts = (snapshot.tasks ?? []).flatMap((t: any) => t.interrupts ?? [])

  if (interrupts.length > 0) {
    const raw = interrupts[0].value

    // 识别 confirmNode 的结构化中断
    if (raw && typeof raw === 'object' && (raw as any).type === 'confirm') {
      const data = raw as { type: 'confirm'; question: string; suggestions: any[] }
      return {
        interrupted: true as const,
        question: data.question,
        suggestions: data.suggestions ?? [],
        reply: '',
        intent: 'apply',
        documentText: '',
      }
    }

    // 普通字符串中断（askForMoreNode）
    const question = typeof raw === 'string' ? raw : String(raw)
    return {
      interrupted: true as const,
      question,
      suggestions: [],
      reply: '',
      intent: (snapshot.values as any)?.intent ?? 'insufficient',
      documentText: (snapshot.values as any)?.documentText ?? '',
    }
  }
  return null
}
```

#### 改动 3：`agentService.ts` — `streamAgent` / `streamResume` 直接使用 interruptResult.suggestions

```typescript
// 修改前（两处）
const snapshot = await app.getState(config)
const state = snapshot.values as any
const suggestions = (state.checkResults ?? [])...

yield {
  type: 'interrupt',
  data: {
    question: interruptResult.question,
    suggestions,                          // ← 这里用的是从主图读的空数组
    requireFiles: suggestions.length > 0,
  }
}

// 修改后（两处）：直接用 interruptResult.suggestions
yield {
  type: 'interrupt',
  data: {
    question: interruptResult.question,
    suggestions: interruptResult.suggestions,              // ← 来自中断值
    requireFiles: interruptResult.suggestions.length > 0, // ← 正确判断
  }
}
// 同时删除 const snapshot2/state2/suggestions2 的冗余读取
```

## 数据流对比

### Bug 3 修复前后对比

```
修复前：
confirmNode → interrupt("已匹配...") → checkInterrupt → state.checkResults = [] → suggestions=[] → requireFiles=false
→ onInterrupt else分支 → aiInterrupted=true → 文字补充框（无按钮）

修复后：
confirmNode → interrupt({type:'confirm', question:"已匹配...", suggestions:[...]}) 
→ checkInterrupt 识别 type=confirm → suggestions=[{...}] → requireFiles=true
→ onInterrupt if分支 → push confirm-action消息 → 内联显示按钮✅
```

### Bug 2 修复前后对比

```
修复前：
streamResume → ask节点重放 → LLM生成追问tokens → [用户看到重复回答] → interrupt返回 → applyGraph → summarize tokens → confirm interrupt

修复后：
streamResume → ask节点重放 → LLM tokens [被SKIP_NODES='ask'过滤] → interrupt返回 → applyGraph → summarize tokens [被SKIP_NODES='summarize'过滤] → confirm interrupt
→ onInterrupt with suggestions → confirm-action气泡✅
```
