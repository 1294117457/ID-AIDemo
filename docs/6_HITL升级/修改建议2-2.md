### B. LLM 感知历史的追问

#### `prompts.ts` 新增函数

```ts
/**
 * 生成感知历史的追问文本（用于 askForMoreNode）
 * 输出：一句≤60字的中文追问，告知已记录了什么，询问还缺什么
 */
export function contextualAskPrompt(allUserText: string, missingFields: string[]): string {
  return `你是一个申请助手，请根据用户已有输入，生成一句简洁友好的追问。
要求：
- 告知用户已经记录了哪些信息（赛事/等级/时间/角色中已知的部分）
- 询问仍然缺少的字段：${missingFields.join('、')}
- 不超过60字，语气亲切，不重复已有内容

用户已有输入历史：
${allUserText}

请直接输出追问文本，不要加任何前缀。`
}
```

#### `mainGraph.ts` 中 `askForMoreNode` 修改

```ts
// 现在（硬编码模板）：
const question = `申请材料不完整，还缺：${state.missingInfo.join('、')}。请补充：`

// 修改后（LLM 生成）：
const allUserText = state.messages
  .filter(m => m instanceof HumanMessage)
  .map(m => String(m.content))
  .join('\n')
const model = createChatModel(0)
const reply = await model.invoke([new HumanMessage(contextualAskPrompt(allUserText, state.missingInfo))])
const question = String(reply.content)
```

> **注意**：此调用 temperature=0，prompt 约 200 tokens，每次追问额外消耗 1 次模型请求。
> 后续可在 `getContextMaxMessages` 限制下控制总轮数。

```ts
  role: 'user' | 'assistant' | 'confirm-action'
```

#### `onInterrupt` 回调修改

```ts
// 修改前：当 requireFiles=true 时，仅设置 aiInterrupted + aiInterruptSuggestions
// 修改后：同时向 aiMessages 推入 confirm-action 消息
onInterrupt(question, extra) {
  aiLoading.value = false
  aiStreaming.value = false
  aiInterrupted.value = true
  aiMessages.value[aiMsgIdx].content = question   // AI 气泡显示匹配摘要

  if (extra?.requireFiles) {
    aiInterruptSuggestions.value = extra.suggestions ?? []
    // 在消息流中插入操作气泡
    aiMessages.value.push({ role: 'confirm-action', content: '', done: false })
  }
  scrollAiToBottom()
},
```

```ts
const handleAiConfirm = () => {
```

#### G4. `agentService.ts` 检查上下文上限

在 `streamAgent` 函数开头新增：

```ts
export async function* streamAgent(input: AgentInput): AsyncGenerator<...> {
  const config = { configurable: { thread_id: input.sessionId } }
  const app = await getApp()

  // ── 上下文长度检查 ──
  const maxMessages = getContextMaxMessages()
  const currentState = await app.getState(config)
  const currentMsgCount = (currentState?.values as any)?.messages?.length ?? 0
  if (currentMsgCount >= maxMessages) {
    yield {
      type: 'error',
      data: { message: `对话上下文已达上限（${maxMessages} 条消息），请点击「开始新对话」清空后继续。` }
    }
    return
  }

  // ...原有 streamEvents 逻辑...
}
```

## 三、数据流变化（对比）

### 上下文超限流程（新增）

```
用户连续对话 N 轮（N >= contextMaxMessages）
  ↓
agentService.streamAgent():
  app.getState(config) → messages.length >= maxMessages
  yield { type: 'error', data: { message: '上下文已达上限，请开始新对话' } }
  ↓
前端 onError 回调：显示错误气泡
建议文字：「对话已超出长度限制，请点击『开始新对话』」
```

### 确认提交流程（优化后）

```
confirmNode → interrupt(简洁摘要)
  ↓
agentService.streamAgent():
  yield { type: 'interrupt', data: { question, suggestions, requireFiles: true } }
  ↓
前端 onInterrupt:
  AI 气泡显示摘要文字
  aiMessages.push({ role: 'confirm-action', done: false })
  ↓
用户上传文件 + 点击「确认提交」
handleAiConfirm():
  confirm-action 消息 done = true  → 气泡显示「✅ 已操作」
  agentResumeStream(sessionId, '{"action":"confirm","proofFileIds":[...],"proofValues":[...]}')
  ↓
submitNode 解析 parsed.action === 'confirm' → 提交到 Java → 返回申请编号
```

## 五、待确认事项

1. **上下文上限默认值**：建议默认 **20 条**（约 10 轮对话），是否合适？
   - 加分申请流程：`classify→ask→classify→apply→confirm→submit` 约 8-12 条消息，20 条足够完成一次申请。
   - 如果允许更长的咨询对话，可调高至 40。

2. **上下文超限后的旧对话保留**：超限后旧 checkpoint 数据留在 `agent.db` 中（按 `thread_id` 隔离）。
   是否需要定期清理？目前不影响使用，可后续再加 cron 清理任务。

3. **引导提示的显示策略**：建议默认**展开**（新用户友好），折叠后用 `localStorage` 记住状态，
   下次同一用户打开时不再展开。是否同意？

4. **`confirm-action` 气泡宽度**：`FileTable` 组件在 `confirm-action` 气泡内渲染，
   建议气泡宽度设为 `w-full max-w-none`（撑满对话框内容区），避免 FileTable 被截断。
