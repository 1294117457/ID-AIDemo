# 修改建议 2 — AI 智能申请 UX 优化

> 基于当前代码状态（`applyGraph.ts` / `mainGraph.ts` / `score/index.vue`）进行分析，
> 本建议不含代码修改，仅说明问题、方案和改动范围，供确认后实施。

---

## 一、现状与问题分析

### 1.1 对话框缺少引导
当前打开 AI 智能申请对话框时，只显示：
> 🤖 请上传证书文件或描述您的加分情况，我来帮您匹配

用户不清楚该描述哪些内容，容易因信息不完整导致多轮追问。

---

### 1.2 信息不足时的补充体验

**现有机制（已正确，但体验可优化）：**

```
用户: "我得了三等奖"
  → classifyNode: intent=insufficient, missing=[赛事名称, 时间, 等级, 角色]
  → askForMoreNode: interrupt("申请材料不完整，还缺：赛事名称、时间、等级、申请人角色。请补充：")
  → 前端: 展示文字输入框等待补充
用户补充: "挑战杯，2024年，个人"
  → streamResume → Command({resume: supplement})
  → askForMoreNode 返回 → 重回 classify
  → classify 读取 ALL HumanMessages 历史 → 信息仍不足 → 再次 interrupt
```

LangGraph 的 `SqliteSaver` 已正确保存对话历史，`classifyNode` 使用全部 `HumanMessage` 历史重新分析，**不需要用户重复输入**之前说过的内容。

**问题所在（体验层面）：**
1. `askForMoreNode` 的追问只列出缺失字段，不告知用户"已经记住了什么"，用户体验不友好
2. `askForMorePrompt` 是固定模板 `"申请材料不完整，还缺：xxx。请补充："` 无语境感知
3. 前端的文字补充输入框样式不够突出，与普通输入框一样，用户不知道这是"系统在等待补充"

---

### 1.3 确认交互要求用户输入 JSON

`confirmNode` 当前的 interrupt question（即 AI 气泡内容）包含：
```
请通过页面上传对应的证明材料，获取文件 ID 后，回复以下 JSON 确认提交：
{"confirm":true,"proofFileIds":[文件ID1],"proofValues":[分值1]}
或回复 cancel 取消申请。
```

这对用户完全不友好。前端 `handleAiConfirm` 已经通过 `FileTable` + 按钮采集数据，
在内部组装 JSON 传给 `agentResumeStream`，**技术上已正确**。

但问题是：
1. AI 气泡内的文字还在显示 JSON 指令，污染对话 UI
2. 确认/取消按钮在消息列表**外侧**底部，视觉上与对话流割裂
3. `submitNode` 内有一大段 JSON 解析逻辑（本可简化）

---

## 二、修改方案

---

### 修改点 1：对话框引导提示

**目标**：用户打开对话框时，清晰了解可提供哪些信息。

**方案**：在消息列表**顶部**（messages 为空时始终显示）增加一个引导卡片：

```
🤖  AI 智能申请助手

  您可以通过以下方式发起申请：

  ① 上传证书文件（PDF / Word）— 点击 📎 选择文件
  ② 直接描述比赛情况，需包含以下 4 项：
      • 赛事名称（如：挑战杯、互联网+）
      • 奖项等级（如：国家级一等奖、省级三等奖）
      • 获奖时间（如：2024年5月）
      • 您的角色（如：个人、队长、队员）

  示例："2024年11月参加全国大学生数学建模，获国家级二等奖，我是队长"
```

**改动文件**：`idfrontend/src/views/score/index.vue`（仅模板部分，无逻辑改动）

---

### 修改点 2：信息不足时的连续补充体验

#### 2a. Agent 端：追问消息感知历史

**目标**：追问时告知用户已知信息，只问缺失部分。

**当前 `askForMoreNode`**（`mainGraph.ts` 第 54-65 行）：
```ts
async function askForMoreNode(state) {
  const question = `申请材料不完整，还缺：${state.missingInfo.join('、')}。请补充：`
  const userAnswer = interrupt(question)
  ...
}
```

**建议改为**：调用 LLM 生成一句感知历史的追问，替换硬编码模板。
新增一个 `contextualAskPrompt` 函数，并在 `prompts.ts` 中定义：

```ts
// prompts.ts 新增
export function contextualAskPrompt(allUserText: string, missingFields: string[]): string {
  return `
根据用户已有输入，生成一句简洁友好的追问，告知用户已经记录了哪些信息，
并询问还缺少的字段。不超过60字，语气亲切。

用户已有输入：${allUserText}
仍然缺少：${missingFields.join('、')}
`
}
```

`askForMoreNode` 改为：
```ts
async function askForMoreNode(state) {
  // 1. 用 LLM 生成感知历史的追问（temperature=0）
  const allUserText = state.messages
    .filter(m => m instanceof HumanMessage)
    .map(m => m.content).join('\n')
  const model = createChatModel(0)
  const reply = await model.invoke([new HumanMessage(contextualAskPrompt(allUserText, state.missingInfo))])
  const question = String(reply.content)

  // 2. interrupt 等待用户回复
  const userAnswer = interrupt(question)
  return {
    messages: [new AIMessage(question), new HumanMessage(String(userAnswer))]
  }
}
```

> **数据流不变**：`classifyNode` 仍然读取所有 `HumanMessage` 历史重新判断，
> `SqliteSaver` 已保存全部状态，**用户无需重复输入之前说过的内容**。

---

#### 2b. 前端端：补充输入区视觉强化

**目标**：让用户清楚知道系统正在等待补充，而不是普通输入。

**当前**（信息不足 interrupt 时）：
```html
<!-- HITL：信息不完整时的文字补充 -->
<div v-else-if="aiInterrupted" class="mt-3">
  <div class="flex gap-2">
    <el-input v-model="aiSupplement" placeholder="请补充信息..." ... />
    <el-button>发送</el-button>
  </div>
</div>
```

**建议改为**：补充输入区加背景色 + 说明文字，区分于普通输入：

```html
<!-- 信息不足时的补充输入区 -->
<div v-else-if="aiInterrupted" class="mt-3 rounded-lg bg-amber-50 border border-amber-200 p-3">
  <div class="text-xs text-amber-700 mb-2 font-medium">
    ⚠️ 系统已记住您之前的内容，请继续补充缺少的信息：
  </div>
  <div class="flex gap-2">
    <el-input v-model="aiSupplement" placeholder="继续补充..." @keyup.enter="handleAiResume" />
    <el-button type="warning" @click="handleAiResume" :disabled="!aiSupplement.trim()">发送</el-button>
  </div>
</div>
```

**改动文件**：`idfrontend/src/views/score/index.vue`（仅模板/样式，约 8 行）

### 修改点 3：确认交互改为内联按钮

#### 3a. Agent 端：简化 `confirmNode` 的 interrupt question

**目标**：AI 气泡中只展示匹配结果摘要，不出现 JSON 指令。

**`confirmNode` interrupt question 改为**：

```ts
const question = [
  `已为您匹配到以下加分项，请上传对应证明材料后点击确认提交：`,
  '',
  ...suggestions.map((s, i) =>
    `${i + 1}. **${s.templateName}** / ${s.ruleName}（预计 ${s.estimatedScore} 分）\n   ${s.reason}`
  ),
].join('\n')

// interrupt 只传简洁的 question，前端通过按钮采集数据
const userAnswer = interrupt(question)
```

**`submitNode` 简化 resume 格式**：前端传的 resume 值改为结构化对象（JSON 字符串），
`submitNode` 解析 `{action, proofFileIds, proofValues}`，不再有 JSON 格式说明和错误提示逻辑。

---

#### 3b. 前端端：确认按钮移入消息流（关键改动）

**目标**：确认/取消按钮出现在 AI 消息气泡正下方，融入对话流。

**实现思路**：

新增消息类型 `'confirm-action'`，当 `onInterrupt` 触发且 `requireFiles: true` 时，
向 `aiMessages` 推入两条消息：
1. `{ role: 'assistant', content: 匹配摘要文字 }`（普通 AI 气泡）
2. `{ role: 'confirm-action', content: '' }`（特殊操作气泡）

`confirm-action` 气泡渲染为：

```html
<!-- confirm-action 气泡：跟在 AI 气泡后面 -->
<div v-else-if="msg.role === 'confirm-action'" class="flex justify-start pl-8">
  <div class="max-w-[85%] space-y-3">
    <!-- 文件上传区 -->
    <div class="bg-white border rounded-xl p-3 shadow-sm">
      <div class="text-xs text-gray-500 mb-2">上传证明材料：</div>
      <FileTable v-model="aiProofItems" ... />
    </div>
    <!-- 按钮区 -->
    <div class="flex gap-2">
      <el-button size="small" @click="handleAiCancel">取消申请</el-button>
      <el-button size="small" type="primary"
        :disabled="aiProofItems.length === 0 || aiLoading"
        @click="handleAiConfirm">
        确认提交
      </el-button>
    </div>
  </div>
</div>
```

**同时移除**现有的独立 `v-if="aiInterrupted && aiInterruptSuggestions.length > 0"` 区块。

## 五、需要确认的细节

1. **修改点 2a 是否引入 LLM 调用**：`askForMoreNode` 改用 LLM 生成追问会额外消耗 1 次模型调用（prompt 约 150 tokens）。如果不希望增加消耗，可以保留硬编码模板，只改样式（改动 2b）。

2. **`confirm-action` 气泡位置**：方案是在消息列表内渲染 FileTable，这意味着 FileTable 会进入滚动区域。如果 FileTable 组件本身高度较大，建议将消息列表高度从 `h-80` 改为 `h-[500px]`，或将对话框高度加大。

3. **`FileTable` 的 `v-model`**：`aiProofItems` 仍作为状态变量，`confirm-action` 气泡绑定同一个 `ref`，与现有逻辑兼容。

4. **取消/确认按钮的显示时机**：`confirm-action` 消息只在 `aiInterrupted === true` 时渲染按钮（上传区和按钮），一旦用户确认/取消，立即移除该状态（`aiInterrupted = false`），历史消息中的 `confirm-action` 渲染为空（或只显示"已操作"提示）。
