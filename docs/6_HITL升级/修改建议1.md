# 修改建议 1：入口整合 + HITL + 代码精简

> 纯建议文档，未修改任何代码。请确认方向后再动手。

---

## 一、现状分析

### 当前有两个独立的 AI 入口

**入口 A：悬浮聊天球** (`idfrontend/src/views/ai-agent/index.vue`)
- 纯对话，支持 interrupt HITL（信息不完整时追问）
- 支持上传文件（代码里有 `pendingFile`，前端也有文件上传按钮）
- 调用：`POST /agent/stream` → Java → Node Agent (LangGraph)

**入口 B：加分申请页 AI 智能分析** (`idfrontend/src/views/score/index.vue`)
- 固定三步流程：上传 PDF → 解析推荐 → 确认提交
- 没有对话，没有 HITL（文件解析失败直接弹错误，没有追问）
- 调用：`POST /analyze/certificate` + `POST /analyze/generate` → Java → Node Agent（直接调 LLM，**绕过了 LangGraph**）

### 两个入口的本质区别

| | 聊天球（入口 A） | 加分页（入口 B） |
|---|---|---|
| 交互方式 | 多轮对话 + SSE 流式 | 固定三步弹窗 |
| AI 通道 | LangGraph（applyGraph） | 直接调 LLM（analyze.ts） |
| HITL | 有（信息不足时追问） | 无 |
| 文件上传 | 有（但建议去掉） | 有（核心功能） |
| 提交申请 | 无（只匹配，不提交） | 有（最终调 submitBonusApplication） |

**核心问题：两套 AI 分析逻辑并行存在**（applyGraph 和 analyze.ts 做的事几乎一样），且 B 入口没有 HITL。

## 二、整合方案

### 目标效果

把"加分申请页"的 AI 功能改造成**带 HITL 的对话式流程**，不再用固定三步弹窗，改为在页面内嵌一个对话区域（或复用聊天球组件）。同时让 LangGraph 的 applyGraph 统一处理所有申请分析，废弃独立的 `analyze.ts`。

```
整合后：
                      用户（加分申请页）
                           │  上传 PDF + 对话
                           ▼
              POST /agent/stream（LangGraph 统一入口）
                           │
                      mainGraph
                      classify → apply → applyGraph
                                           ├── fetchPolicy
                                           ├── analyzeAndMatch
                                           ├── summarize
                                           └── confirm(interrupt) ← 新增 HITL
                                                    │
                                             用户确认后调
                                        submitBonusApplication
```

### 具体变更

#### 前端（idfrontend）

**1. `ai-agent/index.vue`（聊天球）**
- 去掉文件上传按钮和 `pendingFile` 相关代码（约 30 行）
- 保留纯文字对话功能
- 聊天球只做咨询（consult），不做申请

**2. `score/index.vue`（加分申请页）**
- 删除现有"AI 智能分析"三步弹窗（`aiDialogVisible`、`handleAiAnalyze`、`handleAiSelectSuggestion`、`handleAiSubmit`，约 120 行 JS + 对应 template）
- 在页面中新增一个内嵌对话区（复用聊天球的 `buildCallbacks` + `consumeSSE` 逻辑），支持：
  - 发送消息
  - 上传 PDF 文件
  - 显示 interrupt UI（材料不足时展示追问输入框）
  - 接收最终匹配结果并展示加分建议
- 对话区可以是一个独立的 Vue 组件，和聊天球共享 API 层

#### Agent（ID-AIDemo）

**3. `applyGraph.ts` 新增 confirmNode（HITL）**

在 `summarize` 节点之后加一个 `confirm` 节点，interrupt 等待用户确认：

```
summarize → confirm(interrupt) → 用户回复"确认" → submitNode → END
                                → 用户回复其他  → cancelNode → END
```

confirm 节点内容：展示匹配结果摘要，问用户"确认提交吗？"

**4. `applyGraph.ts` 新增 submitNode**

用户确认后，Agent 需要通知 Java 后端提交申请。这里有两种方案（见下文第三节）。

**5. 删除 `routes/components/analyze.ts`（整合后废弃）**

功能完全被 applyGraph 覆盖。Java 后端的 `analyzeCertificate` 和 `generateApplication` 两个调用也一并删除。

**6. 同步删除 Java 后端对应方法**（`AIAgentService.java` 中 `analyzeCertificate`、`generateApplication`，`AIAgentController.java` 中 `/api/ai/analyze/certificate`、`/api/ai/analyze/generate`）

#### 非流式路由精简

删除 `/agent/chat` 和 `/agent/resume`（保留流式版本即可），agent.ts 只剩 `/stream` 和 `/resume-stream` 两个接口。

## 三、HITL confirm 后如何提交申请（关键决策点）

这是整合方案中最需要你决策的地方，有两种方案：

### 方案 A：Agent 回调 Java 后端提交（推荐）

confirm 节点用户确认后，Agent 内部调用 Java 后端的提交接口：

```
用户确认 → submitNode → HTTP POST → Java /api/score/submit → 写入 DB
                                         ↓
                               返回申请编号给 Agent
                                         ↓
                          Agent SSE 推送 "提交成功，申请编号 xxx"
```

优点：用户在对话中一步完成，不需要跳转页面。
缺点：Agent 需要知道 Java 后端地址，且需要传递用户信息（studentId 等）— 这些信息需要在 `/agent/stream` 请求时由 Java 后端一并带入 state。

实现：在 `AgentInput` / `MainState` 中新增 `studentInfo` 字段，Java 发起 stream 请求时附带。

### 方案 B：Agent 返回结果，前端弹窗确认后提交（现有流程变体）

Agent 在 summarize 结束后返回匹配结果（result 事件），前端拿到结果后展示确认按钮，用户点确认后前端直接调 `submitBonusApplication`。

```
Agent SSE result → 前端展示匹配结果 + 确认按钮
                           ↓ 用户点击
               前端 POST /api/score/submit → Java → DB
```

优点：Agent 不需要改，提交逻辑全在前端，实现简单。
缺点：这不算真正的 HITL（interrupt 发生在前端，不在图里），更接近现有的"三步弹窗"流程换了个外壳。

**建议选方案 A**，因为你的目标是在 LangGraph 内部实现 HITL confirm，让图真正控制整个流程。如果 studentInfo 传递觉得麻烦，可以先用方案 B 过渡。

## 四、ID-AIDemo 结构优化

### 现状

```
src/
├── agent/
│   ├── mainGraph.ts
│   ├── prompts.ts
│   ├── state.ts
│   └── subGraphs/
│       ├── applyGraph.ts
│       └── consultGraph.ts
├── common/
│   └── upload.ts
├── db/
│   └── init.ts
├── routes/
│   ├── index.ts
│   └── components/
│       ├── agent.ts
│       ├── analyze.ts      ← 整合后删除
│       ├── config.ts
│       └── knowledge.ts
├── services/
│   ├── agentService.ts
│   ├── aiConfig.ts
│   ├── knowledgeManager.ts
│   └── llmService.ts
├── types/
│   └── scoreTemplate.ts    ← 只有两个 interface，可以内联
└── index.ts
```

### 建议

**1. 删除 `src/types/scoreTemplate.ts`，内联到 `state.ts`**

这两个 interface 只被 `state.ts`、`applyGraph.ts`、`analyze.ts`（删除后）、`agent.ts` 用到。整合后使用方减少，直接在 `state.ts` 里定义：

```typescript
// state.ts 顶部
export interface TemplateRule {
  id: number
  ruleName: string
  ruleScore: number
  description?: string
}

export interface ScoreTemplate {
  id: number
  templateName: string
  templateType: string
  scoreType: number
  templateMaxScore?: number
  description?: string
  rules: TemplateRule[]
}
```

其他文件从 `../agent/state.js` import 即可，省掉一个独立文件。

**2. `applyGraph.ts` 里仍有直接 import `aiConfig` 的残留**（第 7 行 `getApiKey/getBaseUrl/getChatModel`），整合后统一走 `llmService.ts`，这几行可以删掉。

**3. `analyze.ts` 里的 `extractJsonArray` 工具函数**，整合后如果 applyGraph 改用 `withStructuredOutput`（已经在用），这个函数就不需要了。

**4. `common/upload.ts` 位置建议**：改叫 `shared/upload.ts` 或 `utils/upload.ts`，`common` 这个名字语义比较模糊。
