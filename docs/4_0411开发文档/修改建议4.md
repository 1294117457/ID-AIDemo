## 问题 1：知识库上传后不同步 + 可视化不足

### 1.1 当前上传链路分析

```
管理端前端 (knowledge/index.vue)
  ↓ POST /api/knowledge/upload (FormData)
Spring Backend (KnowledgeController)
  ↓ OkHttp multipart 转发
Agent (routes/knowledge.ts  POST /knowledge/upload)
  ↓ multer 接收文件到 uploads/
  ↓ callTool('parse_document', {filePath, ext})   ← MCP 子进程解析
  ↓ callTool('ingest_document', {sourceFile, text}) ← MCP 子进程分块+嵌入+入库
  ↓ res.json({ code: 200 })
  ↓ 前端收到 200 → 调用 loadFiles() 刷新列表
Agent (routes/knowledge.ts  GET /knowledge/list)
  ↓ listSources()   ← **主进程**直接查 SQLite
```

### 1.2 可能导致"不同步"的原因

| # | 可能原因 | 说明 |
|---|---------|------|
| A | **MCP 子进程嵌入 API 失败** | `ingest_document` 内部调用 `getEmbeddings(chunks)`，如果 API Key 为空、过期、被限流，整个工具调用 throw → 上传路由返回 500。但如果前端没有正确展示 500 的 msg 字段，用户可能觉得"没同步" |
| B | **跨进程 SQLite 可见性** | MCP 子进程（`knowledgeServer.ts`）和主进程分别各自 `initDb()` 打开同一个 `data/agent.db`。默认 journal mode = "delete"，理论上写入提交后对方可见。但如果某一方开启了 WAL 模式或有缓存，可能出现延迟 |
| C | **上传文件太大 / 嵌入耗时太长** | 20MB PDF + 大量 chunk → 多轮嵌入 API 调用。OkHttp 的 `readTimeout=120s` 或前端 `timeout=120000` 超时后中断，后端认为"失败"但 Agent 侧仍在写入 |
| D | **上传成功但前端刷新时机问题** | `handleUpload` 中 `await loadFiles()` 在 `ElMessage.success()` 之后立即执行，基本不会有问题，但如果 `loadFiles()` 本身因 Agent 不可达而静默失败，列表不会更新 |
| E | **解析为空** | `.doc` 格式（旧版 Word）解析不出内容，`parse_document` 返回空文本 → 上传路由返回 400 但 msg 只是"文件内容为空"，用户可能没注意到 |

## 问题 3：AI 配置的运行机制详解

### 3.1 整体架构

```
管理端前端                   Spring 后端                   Agent (Node.js)
ai-config/index.vue    →    AIConfigController     →    routes/config.ts
修改表单并点击保存          PUT /api/ai-config            PUT /config
                          (OkHttp 透传 JSON)           (aiConfig.ts 写 SQLite)
```

### 3.2 数据存储

**Agent 端 `data/agent.db` 的 `ai_config` 表：**

| config_key | config_value | updated_at |
|-----------|-------------|-----------|
| system_role | 你是厦门大学... | 1712345678 |
| api_key | sk-xxx... | 1712345678 |
| base_url | https://dashscope... | 1712345678 |
| chat_model | qwen3-max | 1712345678 |
| embedding_model | text-embedding-v3 | 1712345678 |

初始化时由 `db/init.ts` 的 `INSERT OR IGNORE` 插入默认值。

### 3.3 读取流程（Agent 如何使用配置）

`aiConfig.ts` 是配置中枢，核心设计是**内存缓存 + 按需加载**：

```
aiConfig.ts
├── _cache: Record<string, string> | null    ← 内存缓存
├── loadAll()                                 ← 若缓存为空则查 SQLite，否则直接返回缓存
├── getSystemRole() → loadAll()['system_role']
├── getApiKey()     → loadAll()['api_key']    （空则回退到环境变量）
├── getChatModel()  → loadAll()['chat_model']
├── getBaseUrl()    → loadAll()['base_url']
├── getEmbeddingModel() → loadAll()['embedding_model']
└── invalidate()    → _cache = null           ← 清空缓存
```

在 Agent 的各个模块中，配置并不是启动时一次性读取的常量，而是**每次创建 LLM 实例时实时调用 getter**。例如 `mainGraph.ts` 的 `classifyNode`：

```typescript
async function classifyNode(state) {
  const classifierModel = new ChatOpenAI({
    apiKey: getApiKey(),              // ← 每次调用都从 aiConfig 读最新值
    configuration: { baseURL: getBaseUrl() },
    modelName: getChatModel(),
  })
  // ...
}
```

同理，`embeddings.ts` 中：

```typescript
export async function getEmbedding(text: string) {
  const API_KEY = getApiKey()         // ← 每次调用都从 aiConfig 读最新值
  const BASE_URL = getBaseUrl()
  const MODEL = getEmbeddingModel()
  // ...
}
```

### 3.4 修改流程（管理员保存配置后发生了什么）

**第 1 步：前端发送**

`ai-config/index.vue` 点击"保存配置"：

```typescript
const update = { systemRole, chatModel, embeddingModel, baseUrl }
if (editingApiKey && form.apiKey) update.apiKey = form.apiKey
await updateAIConfig(update)
// → PUT /api/ai-config  body: { systemRole: "...", chatModel: "qwen3-max", ... }
```

**第 2 步：Spring 后端透传**

`AIConfigController.updateConfig()` → `AIAgentService.updateAIConfig(body)` → OkHttp PUT `http://localhost:3001/config`

后端本身**不存储**任何 AI 配置，只做 HTTP 代理。

**第 3 步：Agent 写入 SQLite + 清缓存**

`routes/config.ts` 的 `PUT /` → 调用 `updateConfig()`：

```typescript
export function updateConfig(update: ConfigUpdate): void {
  // 1. 构建要更新的键值对
  const entries: [string, string][] = []
  if (update.systemRole != null) entries.push(['system_role', update.systemRole])
  if (update.apiKey != null && update.apiKey.trim() !== '')
    entries.push(['api_key', update.apiKey.trim()])
  // ...

  // 2. 事务批量 UPSERT 到 SQLite
  if (entries.length > 0) {
    upsertMany(entries)   // INSERT ... ON CONFLICT ... DO UPDATE
    invalidate()          // ← 关键！清空内存缓存
  }
}
```

**第 4 步：下次对话自动使用新配置**

缓存被 `invalidate()` 清空后，下一次 `getApiKey()` / `getChatModel()` 等被调用时，`loadAll()` 发现 `_cache === null`，重新从 SQLite 读取最新数据。

```
用户发送新消息
  → classifyNode() 内部调用 getApiKey()
  → loadAll() 发现 _cache 为 null
  → 查询 SQLite ai_config 表
  → 返回最新值（刚被管理员修改的）
  → 用新的 apiKey/model 创建 ChatOpenAI 实例
```

### 3.5 关键特性总结

| 特性 | 实现方式 |
|------|---------|
| **即时生效** | `invalidate()` 清缓存 → 下次调用读新值，无需重启 Agent |
| **API Key 安全** | `getConfigView()` 返回掩码 `sk-xx****xxxx`；只有 `updateConfig` 接收明文新值 |
| **环境变量回退** | `getApiKey()` 在 DB 值为空时回退到 `process.env.QWEN3_API_KEY`，保持向后兼容 |
| **配置不跨重启丢失** | 数据在 SQLite，不是纯内存 |
| **后端无状态** | Spring 后端只做 HTTP 代理，不缓存 AI 配置 |

### 3.6 MCP 子进程的配置同步

有一个需要注意的点：MCP 子进程 (`knowledgeServer.ts`) 也 `import` 了 `aiConfig.ts`（通过 `embeddings.ts`）。但 MCP 子进程有**自己独立的内存空间**，它的 `_cache` 和主进程的 `_cache` 是**两份独立的变量**。

当管理员修改配置时，只有**主进程**的 `invalidate()` 被调用。MCP 子进程的缓存**不会被清空**。

**影响**：如果管理员修改了 `embedding_model` 或 `api_key`，MCP 子进程在下次被调用时仍使用旧缓存值，直到缓存自然过期或进程重启。

**建议修复**：在 `aiConfig.ts` 的 `loadAll()` 中加入 TTL（如 60 秒后自动过期），或在 Agent 端的 `PUT /config` 成功后通过某种方式通知 MCP 子进程刷新（如重启子进程）。

最简单的方案是给缓存加 TTL：

```typescript
let _cache: Record<string, string> | null = null
let _cacheTime = 0
const CACHE_TTL = 60_000  // 60 秒

function loadAll(): Record<string, string> {
  if (_cache && Date.now() - _cacheTime < CACHE_TTL) return _cache
  const rows = getDb()
    .prepare('SELECT config_key, config_value FROM ai_config')
    .all() as { config_key: string; config_value: string }[]
  _cache = Object.fromEntries(rows.map(r => [r.config_key, r.config_value]))
  _cacheTime = Date.now()
  return _cache
}
```

这样 MCP 子进程最多 60 秒后就能读到最新配置。
