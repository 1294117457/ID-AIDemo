# 修改建议 3：流式输出混入 JSON + Markdown 渲染

> 对应修改文档3.md 提出的 2 个问题

---

## 零、问题复现

用户发送"挑战杯能加多少分"后，前端显示的 AI 回复内容为：

```
{"intent":"consult","missing":[],"documentText":""}根据《厦门大学信息学院...
```

前面有一段 JSON 是不应该出现的。后面的回答内容是 Markdown 格式（`**粗体**`、`- 列表`、`> 引用`），但前端以纯文本方式渲染，显示原始的 `**` `>` 等标记符号。

---

## 一、问题 1：回复中混入了 classifyNode 的 JSON 输出

### 1.1 根因分析

`streamAgent()` 函数（`mainGraph.ts` 第 193-197 行）使用 `app.streamEvents()` 捕获**所有** LLM 调用的 token：

```typescript
for await (const event of eventStream) {
  if (event.event === 'on_chat_model_stream') {
    const token = event.data?.chunk?.content
    if (token) yield { type: 'token', data: { content: token } }
  }
}
```

一次完整的 consult 流程中有 **2 个 LLM 调用**：

| 节点 | LLM 调用 | 输出内容 | 是否应该流式展示 |
|------|---------|---------|----------------|
| `classifyNode` | `classifierModel.invoke()` (withStructuredOutput) | `{"intent":"consult","missing":[],"documentText":""}` | **否** — 这是分类结果，不是回答 |
| `consultGraph > answerNode` | `model.invoke()` | `根据《厦门大学信息学院...` | **是** — 这是给用户看的回答 |

`streamEvents` 不区分哪个 LLM 产生的 token，所以 classify 的 JSON 和 answer 的回答全部混在一起推给了前端。

同理，apply 流程有 **3 个 LLM 调用**：

| 节点 | 输出 | 是否展示 |
|------|------|---------|
| `classifyNode` | JSON | **否** |
| `applyGraph > analyzeAndMatchNode` | JSON（withStructuredOutput） | **否** |
| `applyGraph > summarizeNode` | 自然语言汇总 | **是** |

### 1.2 修复方案

在 `streamEvents` 循环中，通过 `event.metadata.langgraph_node` 过滤，只推送来自"回答节点"的 token。

**修改文件**：`ID-AIDemo/src/agent/mainGraph.ts`

**修改位置**：`streamAgent()` 和 `streamResume()` 两个函数的 for-await 循环

**修改内容**：

```typescript
// 只流式输出这些节点的 LLM token（consultGraph 的 answer、applyGraph 的 summarize）
const STREAM_NODES = new Set(['answer', 'summarize'])

for await (const event of eventStream) {
  if (event.event === 'on_chat_model_stream') {
    const node = event.metadata?.langgraph_node
    if (!STREAM_NODES.has(node)) continue  // 跳过 classify、analyzeAndMatch 等非回答节点
    const token = event.data?.chunk?.content
    if (token) yield { type: 'token', data: { content: token } }
  }
}
```

**解释**：
- `event.metadata.langgraph_node` 是 LangGraph `streamEvents` v2 提供的元数据，标识产生该事件的图节点名称
- 由于 `consultGraph` 和 `applyGraph` 是子图（subgraph），其内部节点名在 metadata 中仍然保留原始名称（`answer`、`summarize`）
- `classifyNode` 对应的 `langgraph_node` 值为 `classify`；`analyzeAndMatchNode` 的值为 `analyzeAndMatch` — 这些都不在 `STREAM_NODES` 中，因此被跳过

**备选方案**（如果 metadata 不符合预期）：

如果实际运行时 `langgraph_node` 的值不是我们预期的（比如子图节点名带了前缀），可以改用**黑名单方式**：

```typescript
const SKIP_NODES = new Set(['classify', 'classifyNode', 'analyzeAndMatch', 'analyzeAndMatchNode', 'fetchPolicy'])

for await (const event of eventStream) {
  if (event.event === 'on_chat_model_stream') {
    const node = event.metadata?.langgraph_node
    if (node && SKIP_NODES.has(node)) continue
    const token = event.data?.chunk?.content
    if (token) yield { type: 'token', data: { content: token } }
  }
}
```

**调试建议**：在修改前，先在 for-await 循环中加一行日志打印实际的 metadata：

```typescript
console.log('[streamEvents]', event.event, event.metadata?.langgraph_node, event.data?.chunk?.content?.slice(0, 20))
```

根据日志输出确认节点名称，再决定用白名单还是黑名单。

### 1.3 `streamResume()` 同样需要修改

`streamResume()` 函数（第 222-226 行）有完全相同的问题，因为 resume 之后图会重新经过 `classifyNode`。修改方式和 `streamAgent()` 一致。

## 二、问题 2：Markdown 渲染

### 2.1 现状

前端 AI 消息的渲染代码（`index.vue` 第 152 行）：

```html
<div class="whitespace-pre-wrap">{{ msg.content }}</div>
```

使用 Vue 的 `{{ }}` 文本插值，会**原样输出**所有文本，`**粗体**`、`- 列表`、`> 引用` 这些 Markdown 标记都以纯文本形式显示。

### 2.2 推荐方案：`marked` + `DOMPurify`

| 库 | 用途 | 体积 |
|----|------|------|
| `marked` | 将 Markdown 文本转为 HTML | ~40KB (gzip ~14KB) |
| `dompurify` | 清理 HTML 防 XSS 注入 | ~18KB (gzip ~7KB) |

这是最轻量的组合，不引入额外的编辑器或预览组件。

### 2.3 需要改动的文件

#### A. 两端 package.json — 安装依赖

```bash
# idfrontend
cd idfrontend && npm install marked dompurify && npm install -D @types/dompurify

# idfrontend-admin
cd idfrontend-admin && npm install marked dompurify && npm install -D @types/dompurify
```

#### B. 两端 `ai-agent/index.vue` — 渲染改造

**步骤 1：import 库**

在 `<script setup>` 中添加：

```typescript
import { marked } from 'marked'
import DOMPurify from 'dompurify'

// 配置 marked：关闭不需要的特性，提升安全性
marked.setOptions({
  breaks: true,      // 换行符转 <br>（让回复更紧凑）
  gfm: true,         // GitHub Flavored Markdown（支持表格、删除线等）
})

function renderMarkdown(text: string): string {
  if (!text) return ''
  const raw = marked.parse(text) as string
  return DOMPurify.sanitize(raw)
}
```

**步骤 2：修改模板渲染**

将 AI 消息的纯文本输出改为 HTML 渲染：

```html
<!-- 修改前 -->
<div class="whitespace-pre-wrap">{{ msg.content }}</div>

<!-- 修改后 -->
<div class="markdown-body" v-html="renderMarkdown(msg.content)"></div>
```

**涉及 3 处**（两端 index.vue 各 3 处）：
1. AI 消息气泡（`msg.role === 'assistant'` 分支）
2. interrupt 气泡（`msg.role === 'interrupt'` 分支）
3. 流式输出期间的气泡（和第 1 处是同一个 DOM，内容是动态追加的）

#### C. 两端 `ai-agent/index.vue` — 添加 Markdown 样式

在 `<style>` 中添加 Markdown 渲染的基础样式（不用 scoped，因为 `v-html` 生成的元素不受 scoped 约束）：

### 2.4 关于流式渲染的注意事项

流式输出时，`msg.content` 是**逐 token 追加**的。每追加一个 token，`renderMarkdown()` 都会重新解析整段 Markdown。

**潜在问题**：
- 中间状态的 Markdown 可能不完整（如 `**粗体` 只写到一半），`marked` 可能渲染出意外的结构
- 频繁调用 `marked.parse()` 有轻微性能开销

**解决方案（推荐）**：使用 Vue `computed` 或函数缓存，确保渲染不卡顿。由于 `marked.parse()` 本身很快（毫秒级），对于聊天气泡长度的文本不会有性能问题。不完整的 Markdown 中间状态是可接受的（用户能看到内容在"生长"），只要最终完整时渲染正确即可。

如果确实出现闪烁问题，可以在流式进行中暂时用 `whitespace-pre-wrap` 纯文本显示，流结束后再切换为 Markdown 渲染：

```html
<div v-if="isStreaming && index === messages.length - 1" class="whitespace-pre-wrap">{{ msg.content }}</div>
<div v-else class="markdown-body" v-html="renderMarkdown(msg.content)"></div>
```

但通常不需要，先试试直接全程 Markdown 渲染的效果。

修复前：
```
{"intent":"consult","missing":[],"documentText":""}根据《厦门大学信息学院...
**"挑战杯"全国大学生课外学术科技作品竞赛**...
- 若获得**国家级奖项**...
> 注：以上为团体项目加分值。
```

修复后：
- 没有 JSON 前缀
- **"挑战杯"全国大学生课外学术科技作品竞赛** 显示为粗体
- `- 若获得...` 显示为有序/无序列表
- `> 注：...` 显示为引用块（带左边框灰色样式）
