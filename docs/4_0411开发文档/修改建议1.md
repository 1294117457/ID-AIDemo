# 修改建议 1：MCP 集成 + 后端对接 + 流式输出 + 功能偏移修复

> 对应操作文档.md 的 3 项需求，基于对 ID-AIDemo / idbackend / idfrontend / idfrontend-admin 四个工程的完整检查

---

## 零、当前系统现状快照

```
idfrontend / idfrontend-admin
  ↓ apiAIagent.ts → POST /api/chat/send（非流式，旧接口）
  ↓ 无流式调用代码
  ↓ 无 /agent/* 调用代码
idbackend (Spring Boot :8080)
  ├── ChatController       → /api/chat/send, /stream, /clear
  ├── AIAnalyzeController  → /api/ai/analyze/certificate, /generate
  ├── KnowledgeController  → /api/knowledge/*
  └── AIConfigController   → /api/ai-config
        ↓ AIAgentService (OkHttp)
ID-AIDemo (Express :3001)
  ├── 旧路由：/chat/*, /analyze/*, /knowledge/*, /config
  └── 新路由：/agent/chat, /agent/resume（LangGraph，无流式，interrupt 检测有 bug）
```

**核心问题**：
- 前端只调旧接口 `/api/chat/send`（非流式），新 LangGraph Agent 没有任何调用方触达
- 新 Agent 无流式输出、无申请生成步骤、interrupt 检测有 bug、MemorySaver 不持久

## 一、任务 1：MCP Server 集成

### 1.1 目标

将 `docParser.ts`、`embeddings.ts`、`vectorStore.ts` 包装为 MCP tools，让 LangGraph 图节点通过 MCP 协议调用而非直接函数调用，实现数据层解耦。

### 1.2 新增文件

```
ID-AIDemo/
  src/
    mcp/
      knowledgeServer.ts     ← MCP Server 主文件（暴露 3 个 tools）
      mcpClient.ts           ← MCP Client 单例（供 graph 节点调用）
```

### 1.3 knowledgeServer.ts 思路

使用 `@modelcontextprotocol/sdk` 创建 MCP Server，注册 3 个 tools：

| Tool 名称 | 输入 | 内部逻辑 | 输出 |
|-----------|------|---------|------|
| `search_knowledge` | `query: string, topK: number` | `getEmbedding(query)` → `searchSimilar(vec, topK)` | 检索结果文本 |
| `parse_document` | `filePath: string, ext?: string` | `parseFile(filePath, ext)` | 解析后纯文本 |
| `ingest_document` | `sourceFile: string, text: string` | `chunkText()` → `getEmbeddings()` → `saveChunk()` | 入库结果 |

注意事项：
- Server 启动时需要先调用 `initDb()` 确保 SQLite 表存在
- 使用 `StdioServerTransport`（同机子进程模式，最简单）
- 需要在 knowledgeServer.ts 顶部 `import 'dotenv/config'` 加载环境变量

### 1.4 mcpClient.ts 思路

使用 `@langchain/mcp-adapters` 的 `MultiServerMCPClient` 连接到上面的 MCP Server：

```typescript
const client = new MultiServerMCPClient({
  knowledge: {
    transport: "stdio",
    command: "node",
    args: ["--import", "tsx", "src/mcp/knowledgeServer.ts"]
  }
})
```

导出一个 `getKnowledgeTools()` 函数，返回 LangChain Tool 对象数组。

### 1.5 需要新增的依赖

```bash
npm install @modelcontextprotocol/sdk @langchain/mcp-adapters
```

### 1.6 图节点改造

改造 `consultGraph.ts` 的 `retrieveNode` 和 `applyGraph.ts` 的 `fetchPolicyNode`：

**之前**（直接调用函数）：
```typescript
const queryVec = await getEmbedding(query)
const chunks = searchSimilar(queryVec, 5)
```

**之后**（通过 MCP Client 调用 tool）：
```typescript
import { getMcpClient } from '../../mcp/mcpClient.js'

const client = getMcpClient()
const result = await client.callTool("search_knowledge", { query, topK: 5 })
const retrievedContext = result.content[0].text
```

同理，`routes/knowledge.ts` 的上传入库流程也改为调用 `ingest_document` tool。

### 1.7 不动的部分

- `routes/knowledge.ts` 的 HTTP 接口定义不变（GET /list、DELETE /:source 等）
- `services/aiConfig.ts` 不经过 MCP
- `services/analyzeChain.ts` 旧路由仍直接调函数（等旧路由废弃后再改）

## 二、任务 2：Agent 与后端正常交互 + 流式输出

### 2.1 当前调用链路分析

```
前端 apiAIagent.ts
  → POST /api/chat/send（非流式）
    → idbackend ChatController.chat()
      → AIAgentService.chat()
        → POST http://localhost:3001/chat/send（旧 ragChain）

前端没有调用 /api/chat/stream，也没有调用新 /agent/* 接口
```

### 2.2 Agent 端需要做的

#### A. 给新 /agent/chat 加流式支持

新增路由 `POST /agent/stream`，或在 `mainGraph.ts` 中新增 `streamAgent()` 函数：

- 对于 consult 意图：`consultSubgraph` 的 `answerNode` 使用 `model.stream()` 逐 token 输出
- 对于 apply 意图：applySubgraph 包含多个 LLM 调用，不适合逐 token 流式，改为**节点级流式**（每完成一个节点推送一次进度）
- LangGraph 提供 `.streamEvents()` API，可以捕获每个节点的开始/结束事件

具体方案：在 `routes/agent.ts` 新增：

```
POST /agent/stream
  → 参数和 /agent/chat 一致
  → 使用 app.streamEvents() 代替 app.invoke()
  → SSE 格式输出：
    - data: {"type":"node_start","node":"classify"}
    - data: {"type":"token","content":"..."}（consult 时逐 token）
    - data: {"type":"node_end","node":"consultGraph","result":{...}}
    - data: {"type":"interrupt","question":"..."}
    - data: [DONE]
```

#### B. 修复 interrupt 检测（08检查文档 Bug 1）

`mainGraph.ts` 的 `invokeAgent()` 改为用 `app.getState()` 检测 interrupt，不再依赖 try/catch。

#### C. 旧路由保留不动

`/chat/send`、`/chat/stream`、`/analyze/certificate`、`/analyze/generate` 全部保留，确保 idbackend 现有调用不受影响。

### 2.3 idbackend 端需要做的

#### A. AIAgentService.java 新增 3 个方法

| 方法 | 调用 Agent 路由 | 说明 |
|------|---------------|------|
| `agentChat()` | POST /agent/chat | multipart/form-data，支持文字+文件+模板 |
| `agentStream()` | POST /agent/stream | SSE 流式，类似现有 `streamChat()` |
| `agentResume()` | POST /agent/resume | JSON，恢复 interrupt |

`agentStream()` 的实现模式和现有 `streamChat()` 完全一致（OkHttp 读 SSE → SseEmitter 透传），但需要处理新的 SSE 数据格式（多了 `type` 字段）。

#### B. 新建 AIAgentController.java

```java
@RestController
@RequestMapping("/api/agent")
public class AIAgentController {

    @PostMapping("/chat")     // 非流式统一入口
    @PostMapping("/stream")   // 流式统一入口
    @PostMapping("/resume")   // interrupt 恢复
}
```

- `/api/agent/chat` 和 `/stream` 都需要支持 multipart/form-data（有可选 file 参数）
- 在 Controller 内从 MySQL 查模板后传给 AIAgentService
- 返回给前端时需要统一封装为 `ResultVo`

#### C. ChatController 保持不动

旧的 `/api/chat/send`、`/stream`、`/clear` 全部保留。

### 2.4 前端需要做的

#### A. idfrontend（学生端）

**apiAIagent.ts** 需要新增：

```typescript
// 流式对话（通过新 Agent）
export function agentStreamChat(message: string, file?: File): ReadableStream

// interrupt 恢复
export const agentResume = async (supplement: string): Promise<ApiResponse>
```

**ai-agent/index.vue** 需要改造：

当前 `handleSend()` 调用 `sendMessage()`（POST /api/chat/send，非流式，等完整回复）。
改为调用 `agentStreamChat()`，逐 token 追加到 `messages[aiMsgIndex].content`。

需要处理 3 种 SSE 事件：
1. `token` → 追加到当前 AI 消息
2. `interrupt` → 显示补充信息的 UI（可以是一个特殊的消息气泡 + 输入框）
3. `[DONE]` → 结束加载状态

#### B. idfrontend-admin（管理端）

和学生端的 ai-agent 组件完全一样的改造。两个前端的 `apiAIagent.ts` 和 `ai-agent/index.vue` 基本是复制粘贴的关系。

## 三、任务 3：功能偏移修复

### 3.1 D1：对话历史不持久化

**问题**：MemorySaver 纯内存，重启丢失。

**修复方案**：将 MemorySaver 替换为自定义 SQLite Checkpointer。

LangGraph 支持自定义 `BaseCheckpointSaver`，可以用现有的 `better-sqlite3` 实现一个：
- 表结构：`checkpoints(thread_id TEXT, checkpoint_id TEXT, data TEXT, parent_id TEXT, created_at INTEGER)`
- 实现 `getTuple()`、`list()`、`put()`、`putWrites()` 4 个方法

或者使用官方包 `@langchain/langgraph-checkpoint-sqlite`（如果可用）。

**改动文件**：`mainGraph.ts` 第 14 行 `const checkpointer = new MemorySaver()` → 换为 SQLite checkpointer。

### 3.2 D2：无流式输出

**修复方案**：见任务 2.2.A，新增 `/agent/stream` 路由。

### 3.3 D3：无"申请生成"步骤

**问题**：旧流程是 2 步（certificate → generate），新 Agent 只做了第 1 步。

**修复方案（两选一）**：

**方案 A：保留旧 /analyze/generate 不变（推荐，最小改动）**

新 Agent 的 applySubgraph 返回 suggestions 后，前端用新接口获取推荐列表，但用户选择后仍调旧接口 `/api/ai/analyze/generate` 来生成 remark。这条路径完全不受影响。

```
新流程：
  POST /api/agent/chat (带文件) → { reply, intent, suggestions }
  用户选择一项 →
  POST /api/ai/analyze/generate → { remark, applyScore, ... }（走旧路径）
```

**方案 B：在 applySubgraph 加 generateNode**

在 `applyGraph.ts` 的 `summarize` 节点之后加一个 `generateNode`，但这需要用 `interrupt()` 等待用户选择，流程会变复杂。

**建议先用方案 A**，简单直接，不改旧代码。

### 3.4 D4：返回格式差异

**问题**：新 `/agent/chat` 返回 `{ reply, intent, suggestions }` 而非旧的 `{ certificateText, suggestions }`。

**修复**：在 `mainGraph.ts` 的 `invokeAgent()` 返回中补上 `documentText` 字段：

```typescript
return {
  reply: ...,
  intent: result.intent,
  documentText: result.documentText,   // ← 加上这个
  suggestions: ...
}
```

这样前端和 idbackend 可以用 `documentText` 去调旧的 `/analyze/generate`。

### 3.5 D5：无会话清除 API

**修复**：在 `routes/agent.ts` 加一个 `POST /agent/clear`：

```typescript
router.post('/clear', async (req, res) => {
  const { sessionId } = req.body
  // 删除 checkpointer 中对应 thread_id 的记录
  // 如果用 SQLite checkpointer：DELETE FROM checkpoints WHERE thread_id = ?
})
```

## 六、风险点

| 风险 | 说明 | 应对 |
|------|------|------|
| `@langchain/langgraph` v1.x 的 streamEvents API 行为 | 不同版本 API 细节可能不同 | 先写 demo 验证 streamEvents 输出格式 |
| MCP stdio 子进程在 Windows 的兼容性 | stdio transport 在 Windows 上偶尔有编码问题 | 先在本地验证；如有问题改用 HTTP transport |
| SQLite Checkpointer 并发写入 | better-sqlite3 是同步的，多线程安全但多进程不安全 | Agent 只有一个进程，不是问题 |
| 旧接口和新接口并存期间的 sessionId 不互通 | 旧 ragChain 用 SQLite conversations，新 Agent 用 checkpointer，两套独立 | 文档说明：新旧接口的 sessionId 是独立的 |
