## 一、本次修复内容

### 1.1 Bug：前端成功获取 SSE 响应但无法渲染

**现象**：用户在前端发送消息后，Agent 后台日志显示请求正常处理完毕（classify → consult → retrieve → answer 全部执行），但前端 AI 回复气泡始终为空，loading 动画无法消失。

**根因分析**：

调用链路为：

```
前端 agentStreamChat()
  → POST /api/agent/stream (idbackend :8080)
    → AIAgentController.stream()
      → AIAgentService.agentStream()
        → POST /agent/stream (Agent :3001, Express)
          → Agent 返回 SSE: "data: {\"type\":\"token\",...}\n\n" (冒号后有空格)
      ← idbackend 读取行: line.startsWith("data: ") → 提取 JSON payload
      ← Spring SseEmitter 输出: "data:{\"type\":\"token\",...}\n\n" (冒号后无空格)
  ← 前端接收到 Spring 格式的 SSE
```

**关键差异**：

| 来源 | SSE 行格式 | 示例 |
|------|-----------|------|
| Agent (Express `res.write`) | `data: {...}\n\n` | `data: {"type":"token","data":{"content":"根"}}` |
| idbackend (Spring `SseEmitter`) | `data:{...}\n\n` | `data:{"type":"token","data":{"content":"根"}}` |

前端 `consumeSSE` 函数原来的判断：

```typescript
// 修复前 — 要求冒号后必须有空格，Spring 格式不匹配
if (!trimmed.startsWith('data: ')) continue
const payload = trimmed.slice(6).trim()
```

`data:xxx`（无空格）无法通过 `startsWith('data: ')` 检查，**所有 SSE 事件被跳过**，导致 `onToken` 回调从未触发，AI 气泡始终为空。

**修复**：

```typescript
// 修复后 — 兼容 "data: xxx" 和 "data:xxx" 两种格式
if (!trimmed.startsWith('data:')) continue
const payload = trimmed.replace(/^data:\s*/, '').trim()
```

**修改文件**（2 个）：
- `idfrontend/src/api/components/apiAIagent.ts` 的 `consumeSSE` 函数
- `idfrontend-admin/src/api/components/apiAIagent.ts` 的 `consumeSSE` 函数

## 二、MCP 实现详解

### 2.1 什么是 MCP？

**MCP（Model Context Protocol）** 是 Anthropic 发布的开放协议，用于标准化 AI 应用与外部工具/数据源之间的通信。可以理解为 AI Agent 的"USB 接口标准"——不同的工具按照同一个协议接入，Agent 无需关心工具的内部实现。

### 2.2 为什么要在本项目中使用 MCP？

**改造前**：LangGraph 图节点直接调用 `getEmbedding()`、`searchSimilar()`、`parseFile()` 等函数。

```
consultGraph.retrieveNode
  → 直接 import getEmbedding from embeddings.ts
  → 直接 import searchSimilar from vectorStore.ts
```

**问题**：数据层（向量库、文档解析）和 Agent 逻辑强耦合。如果将来换 Qdrant 替代 SQLite 向量库，需要改图节点代码。

**改造后**：数据层操作包装成 MCP Server 暴露的 tools，图节点通过 MCP Client 调用。

```
consultGraph.retrieveNode
  → callTool('search_knowledge', { query, topK: 5 })
    → MCP Client → stdio → MCP Server
      → getEmbedding() + searchSimilar()（内部实现）
```

**好处**：图节点只知道"调用 search_knowledge 工具"，不关心底层实现。换向量库只改 MCP Server。

### 2.4 涉及的文件和职责

#### 2.4.1 MCP Server — `src/mcp/knowledgeServer.ts`

使用 `@modelcontextprotocol/sdk` 创建一个 MCP Server，注册 5 个 tools：

| Tool 名称 | 功能 | 输入参数 | 内部调用 |
|-----------|------|---------|---------|
| `search_knowledge` | 语义检索知识库 | `query: string, topK: number` | `getEmbedding()` → `searchSimilar()` |
| `parse_document` | 解析上传的文档 | `filePath: string, ext?: string` | `parseFile()` |
| `ingest_document` | 分块+嵌入+入库 | `sourceFile, text, chunkSize?, overlap?` | `chunkText()` → `getEmbeddings()` → `saveChunk()` |
| `list_knowledge_sources` | 列出已入库文件 | 无 | `listSources()` |
| `delete_knowledge_source` | 删除指定文件的分块 | `sourceFile: string` | `deleteBySourceFile()` |

**通信方式**：`StdioServerTransport`（标准输入输出），作为 Agent 主进程的**子进程**运行。

**核心代码结构**：

```typescript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'

const server = new McpServer({ name: 'knowledge-server', version: '1.0.0' })

// 注册 tool — 每个 tool 有名称、描述、zod schema 定义的参数、处理函数
server.tool(
  'search_knowledge',                          // tool 名称
  '在知识库中语义检索相关文档片段',               // 描述（LLM 可读）
  { query: z.string(), topK: z.number() },     // 参数 schema（用 zod 定义）
  async ({ query, topK }) => {                  // 处理函数
    const embedding = await getEmbedding(query)
    const chunks = searchSimilar(embedding, topK)
    return { content: [{ type: 'text', text: '...' }] }  // MCP 标准返回格式
  }
)

// 启动：通过 stdio 与 Client 通信
const transport = new StdioServerTransport()
await server.connect(transport)
```

#### 2.4.2 MCP Client — `src/mcp/mcpClient.ts`

使用 `@langchain/mcp-adapters` 的 `MultiServerMCPClient` 连接 MCP Server：

```typescript
import { MultiServerMCPClient } from '@langchain/mcp-adapters'

const client = new MultiServerMCPClient({
  knowledge: {                           // 服务器名称（可配多个）
    transport: 'stdio',                  // 通信方式
    command: 'node',                     // 启动 Server 的命令
    args: ['--import', 'tsx', 'src/mcp/knowledgeServer.ts'],  // 参数
  },
})
```

**关键特性**：
- **懒初始化**：首次调用 `callTool()` 时才启动 MCP Server 子进程
- **单例模式**：全局只有一个 Client 实例，所有图节点共享
- **自动转换**：`client.getTools()` 返回 LangChain `StructuredToolInterface[]`，可以直接用于 LangChain Agent

**导出 3 个函数**：

| 函数 | 用途 | 调用方 |
|------|------|-------|
| `getKnowledgeTools()` | 获取所有 MCP tools（LangChain Tool 对象） | 将来可给 createReactAgent 用 |
| `callTool(name, input)` | 按名称调用单个 tool，返回字符串 | 图节点、路由 |
| `closeMcpClient()` | 关闭 Client，杀死 Server 子进程 | 进程退出时 |

#### 2.4.3 调用方 — 图节点和路由

**consultGraph.ts 的 retrieveNode**（咨询时检索知识库）：

```typescript
import { callTool } from '../../mcp/mcpClient.js'

async function retrieveNode(state) {
  const query = String(userMsg.content)
  const retrievedContext = await callTool('search_knowledge', { query, topK: 5 })
  return { retrievedContext }
}
```

**applyGraph.ts 的 fetchPolicyNode**（申请时检索政策）：

```typescript
async function fetchPolicyNode(state) {
  const policyContext = await callTool('search_knowledge', {
    query: state.documentText.slice(0, 512),
    topK: 5,
  })
  return { policyContext }
}
```

**routes/knowledge.ts 的上传入库**：

```typescript
// 解析文档
const parseResult = await callTool('parse_document', { filePath, ext })
// 分块+嵌入+入库
const ingestResult = await callTool('ingest_document', { sourceFile: fileName, text })
```

### 2.5 MCP 通信流程（以咨询为例）

```
1. 用户发送 "挑战杯能加多少分"
2. mainGraph → classifyNode → intent: "consult"
3. → consultSubgraph → retrieveNode
4. retrieveNode 调用 callTool('search_knowledge', { query: '挑战杯能加多少分', topK: 5 })
5. mcpClient.ts: 找到名为 search_knowledge 的 LangChain Tool 对象
6. Tool.invoke() → 通过 stdio 管道发送 JSON-RPC 请求给 knowledgeServer.ts 子进程
7. knowledgeServer.ts:
   a. getEmbedding('挑战杯能加多少分') → 调 DashScope API 获取向量
   b. searchSimilar(vector, 5) → 在 SQLite 中余弦相似度检索
   c. 返回: { content: [{ type: 'text', text: '[1] (加分政策.md) ...' }] }
8. stdio 管道返回 → mcpClient 解析 → callTool 返回字符串
9. retrieveNode 得到 retrievedContext，写入 state
10. → answerNode → LLM 结合检索结果回答用户
```
