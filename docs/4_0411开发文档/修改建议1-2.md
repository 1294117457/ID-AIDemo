## 一、反馈分析：哪些改了，哪些不变

### 1.1 你的理解完全正确

项目里有两类数据，持久化需求完全不同：

| 数据类型 | 存储位置 | 是否需要持久化 | 当前状态 |
|----------|---------|--------------|---------|
| **知识库**（政策文档 → 分块 → 向量） | SQLite `knowledge_chunks` 表 | **必须持久化**，管理员上传一次要一直可用 | 已持久化，正常工作 |
| AI 配置（api_key、model 等） | SQLite `ai_config` 表 | 必须持久化 | 已持久化，正常工作 |
| **对话上下文**（用户这轮聊了什么） | MemorySaver（内存） | **不需要持久化**，关闭页面即丢弃 | MemorySaver 正好满足需求 |
| LangGraph checkpoint（interrupt 恢复用） | MemorySaver（内存） | 不需要跨重启，单次会话内够用 | MemorySaver 正好满足需求 |

**结论：MemorySaver 完全够用，不需要换 SQLite Checkpointer。**

### 1.2 修改建议1 中哪些条目取消

| 原建议编号 | 内容 | 状态 | 原因 |

## 二、修订后的完整方案

### 2.1 旧代码清理

MemorySaver 意味着对话不持久化，那旧的 `ragChain.ts` 里往 SQLite `conversations` 表存对话历史的逻辑就变成了冗余。有两种处理方式：

**方案 A（推荐，现阶段）：旧路由保留不动**

`/chat/send` 和 `/chat/stream` 仍然走 `ragChain.ts`，它的 SQLite 对话历史机制继续工作。这是旧前端正在用的路径。等新 Agent 路由完全上线后，再考虑废弃。

**方案 B（将来）：废弃旧聊天路由**

当新 `/agent/stream` 完全替代 `/chat/stream` 后：
- 删除 `routes/chat.ts`
- 删除 `services/ragChain.ts`
- `init.ts` 里的 `conversations` 表可以保留（不碍事）或删除

**当前执行方案 A，不动旧代码。**

### 2.2 关于 MemorySaver 的一个补充说明

MemorySaver 在单次会话内是正常工作的：

```
用户打开页面 → 前端生成 sessionId（如 "user_42"）
  → POST /agent/chat { sessionId: "user_42", message: "挑战杯能加多少分" }
    → LangGraph 以 thread_id="user_42" 运行
    → MemorySaver 把 checkpoint 存在内存 Map 里
    
  → POST /agent/chat { sessionId: "user_42", message: "我想申请加分" }
    → MemorySaver 读取 thread_id="user_42" 的 checkpoint
    → 图看到之前的 messages，正常处理多轮

  → interrupt → POST /agent/resume { sessionId: "user_42", supplement: "..." }
    → MemorySaver 读取中断点，正常恢复

用户关闭页面 → 前端不再发请求
Agent 重启 → 内存清空，所有 thread 消失（符合需求）
```

唯一注意：**前端每次打开对话窗口应该生成一个新的 sessionId**（或用 `user_{userId}_{timestamp}`），避免复用旧的 thread 导致状态混乱。

## 三、修订后的执行计划

### Phase 1：Agent 端修复（3 步）

| 步骤 | 文件 | 改动内容 |

**Phase 1 产出**：Agent 端 3 个接口可用：
- `POST /agent/chat` — 非流式，interrupt 正常检测
- `POST /agent/stream` — 流式 SSE
- `POST /agent/resume` — interrupt 恢复

可以用 curl 测试：
```bash
# 咨询
curl -X POST http://localhost:3001/agent/chat \
  -F "message=挑战杯能加多少分" \
  -F "sessionId=test_001"

# 流式
curl -N -X POST http://localhost:3001/agent/stream \
  -F "message=挑战杯能加多少分" \
  -F "sessionId=test_002"
```

### Phase 2：idbackend 对接（2 步）

| 步骤 | 文件 | 改动内容 |
|------|------|---------|
| 4 | `AIAgentService.java` | 新增 `agentChat()`、`agentStream()`、`agentResume()` 3 个方法 |
| 5 | 新建 `AIAgentController.java` | `/api/agent/chat`、`/stream`、`/resume` 3 个接口 |

注意点：
- `agentStream()` 的 SSE 透传逻辑和现有 `streamChat()` 完全一致
- `AIAgentController` 里需要从 MySQL 查模板，然后传给 `agentChat()`/`agentStream()`（和 `AIAnalyzeController.certificate()` 的模板查询逻辑一样）
- sessionId 的生成：从 `httpRequest.getAttribute("userId")` 取，拼为 `"agent_" + userId + "_" + timestamp`，确保每次对话是新 thread

### Phase 3：前端对接（2 步）

| 步骤 | 文件 | 改动内容 |
|------|------|---------|
| 6 | idfrontend + idfrontend-admin 的 `apiAIagent.ts` | 新增 `agentStreamChat()`、`agentResume()` |
| 7 | idfrontend + idfrontend-admin 的 `ai-agent/index.vue` | `handleSend` 改流式；新增 interrupt 补充信息气泡 |

前端核心改造点：

**apiAIagent.ts 新增**：
```typescript
/** 流式对话（SSE） */
export function agentStreamChat(
  message: string,
  sessionId: string,
  file?: File
): EventSource 风格的流式读取

/** interrupt 恢复 */
export const agentResume = (sessionId: string, supplement: string): Promise<ApiResponse>
```

**ai-agent/index.vue 改造**：

当前流程：
```
用户点发送 → sendMessage() → 等完整回复 → 显示
```

改为：
```
用户点发送 → agentStreamChat() → 逐 token 追加显示
  如果收到 interrupt → 显示「补充信息」气泡 → 用户输入 → agentResume()
  收到 [DONE] → 结束
```

需要新增的 UI 状态：
- `isInterrupted: boolean` — 是否处于 interrupt 等待状态
- `interruptQuestion: string` — interrupt 的提问内容
- `currentSessionId: string` — 当前会话 ID

sessionId 生成策略：用户**打开对话窗口时**生成一次，**点击「新对话」时**重新生成。这样同一次对话内 sessionId 不变（多轮有效），关闭再打开就是新的。

### Phase 4：MCP 集成（2 步）

| 步骤 | 文件 | 改动内容 |

## 六、关于「申请加分」流程的完整链路确认

用户说"可以通过 agent 进行咨询或者申请加分"，确认新 Agent 覆盖的完整链路：

### 咨询场景
```
用户："挑战杯能加多少分？"
  → classify → intent: consult
  → consultSubgraph → RAG 检索知识库 → LLM 回答
  → 流式返回回答
```
完整覆盖，无遗漏。

### 申请场景（信息完整）
```
用户："我参加了2024年挑战杯全国决赛，获得金奖，我是负责人，帮我申请加分"
  → classify → intent: apply
  → applySubgraph → RAG 检索政策 → 模板匹配 → 汇总
  → 返回 { reply, intent: "apply", suggestions, documentText }

用户在前端选择一项 suggestion →
  → POST /api/ai/analyze/generate（走旧路径）
  → 返回 { remark, applyScore, ruleId, ... }
  → 前端预填申请表单
```
两步串联，第一步走新 Agent，第二步走旧接口。功能完整。

### 申请场景（信息不全）
```
用户："我想申请加分"
  → classify → intent: insufficient, missing: ["赛事名称", "奖项等级", "时间", "申请人角色"]
  → askForMoreNode → interrupt("还缺：赛事名称、奖项等级、时间、申请人角色。请补充：")
  → 前端收到 interrupted 响应，显示补充信息气泡

用户补充："2024年挑战杯，金奖，负责人"
  → POST /agent/resume { supplement: "2024年挑战杯，金奖，负责人" }
  → askForMoreNode 返回 → classify 再次运行
  → 如果还缺 → 再次 interrupt
  → 如果齐了 → intent: apply → 走 applySubgraph
```
多轮补全完整覆盖。

### 申请场景（带文件上传）
```
用户上传 PDF + 消息"帮我分析这个证书"
  → route 层 parseFile 解析 PDF → documentText
  → classify → intent: apply（材料信息从 PDF 中提取）
  → applySubgraph → 匹配模板 → 返回 suggestions
```
文件上传路径也覆盖。
