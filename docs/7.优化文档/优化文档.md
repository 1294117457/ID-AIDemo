## 一、精细化 LLM 返回参数（Zod Schema ↔ 后端 DTO 严格对齐）

### 问题背景

当前 `analyzeAndMatchNode` 的 Zod Schema 和 Java 后端的 `AgentSubmitDto` 字段存在隐式映射，容易因字段名不一致导致提交失败或数据丢失。

### 当前 Schema 与后端 DTO 对照

| Agent（Zod）         | Java（AgentSubmitDto）    | 说明                        |
|----------------------|---------------------------|-----------------------------|
| `templateId`         | 通过 `state.templates` 查 | 不直接提交，用于查完整模板   |
| `templateName`       | `templateName`            | 直接映射                    |
| `ruleId`             | `ruleId`                  | 直接映射                    |
| `ruleName`           | 无                        | 仅用于展示，不提交           |
| `estimatedScore`     | `applyScore`              | 字段名不同！                |
| `reason`             | `remark`（加前缀后）      | 加了 "AI 智能匹配 - " 前缀  |

### 优化建议：Schema 字段和 Java DTO 完全对齐

**修改 `src/nodes/applyNodes.ts` 的 `SuggestionSchema`：**

```typescript
const SuggestionSchema = z.object({
  suggestions: z.array(z.object({
    templateId:     z.number().describe('匹配的加分模板 ID'),
    templateName:   z.string().describe('模板名称，必须与模板列表中的名称完全一致'),
    ruleId:         z.number().describe('匹配的规则 ID，必须与模板 rules 中的 id 完全一致'),
    ruleName:       z.string().describe('规则名称，用于前端展示'),
    applyScore:     z.number().describe('建议申请的加分分值，不超过该规则的 ruleScore'),
    matchReason:    z.string().max(100).describe('一句话匹配理由，不超过100字'),
  }))
})
```

**同步修改 `src/prompts.ts` 的 `analyzeUserPrompt`，字段名与 Schema 保持一致：**

```typescript
// 将 "estimatedScore" 改为 "applyScore"，"reason" 改为 "matchReason"
"applyScore": <建议申请分值，不超过规则 ruleScore，number>,
"matchReason": <一句话匹配理由，不超过100字，string>
```

**同步修改 `src/nodes/applyNodes.ts` 的 `submitNode` 中的字段引用：**

```typescript
applyScore: suggestion.applyScore,           // 不再用 estimatedScore
remark:     `AI 智能匹配 - ${suggestion.matchReason ?? ''}`,
```

### 对 loadTemplatesJson() 的优化建议（Java 侧）

当前 Java 的 `AIAgentController.loadTemplatesJson()` 缺少 `reviewCount` 字段（已在上版修复）。
建议增加以下字段，供 Agent 在 `submitNode` 中使用：

```java

dto.setTemplateName(t.getTemplateName());
dto.setTemplateType(t.getTemplateType());   // CONDITION | TRANSFORM

```

## 二、Zod 失效问题复盘与避免方案

### 历史上发生的两次 Zod 解析失败

#### 失败 1：`response_format` + messages 无 "json" 关键词

**报错：**
```
Error: 400 InternalError.Algo.InvalidParameter: 'messages' must contain the word 'json'
```

**根本原因：**  
使用 `withStructuredOutput(Schema)` 时，LangChain 底层向 Qwen API 发送 `response_format: { type: "json_object" }`。Qwen 要求此时 messages 中必须出现 "json" 字符串，否则拒绝请求。

**修复方案（已应用）：**
1. `ANALYZE_SYSTEM` 加入"以 JSON 格式返回结果"
2. `model.invoke()` 必须传 `SystemMessage`

**今后规范：**
```typescript
// ✅ 正确写法：withStructuredOutput 必须配 SystemMessage 且含 "json"
const model = createChatModel(0).withStructuredOutput(MySchema)
const result = await model.invoke([
  new SystemMessage('...以 JSON 格式返回结果...'),  // ← 必须包含 "json"
  new HumanMessage(userPrompt),
])
```

---

#### 失败 2：LLM 返回了结构不符合 Schema 的 JSON

**报错：**
```
Error: Failed to parse. Text: "{ "id": 20, "templateName": "ACM竞赛", ... }"
Error: [ { "code": "invalid_type", "expected": "array", "received": "undefined",
           "path": ["suggestions"], "message": "Required" } ]
```

**根本原因：**  
模型自行决定返回了一个单对象（字段名还不对），而非 `{ suggestions: [...] }` 包裹的数组。原 prompt 没有明确的输出格式描述，模型"发挥"了。

**修复方案（已应用）：**  
在 `analyzeUserPrompt` 中明确给出完整 JSON 结构模板。

**今后规范：**

```
规范1：prompt 里明确写出期望的 JSON 结构（字段名、类型、嵌套层次）
规范2：Schema 的 .describe() 要准确描述约束，帮助 LLM 理解
规范3：对于可能失败的解析，加 try/catch 降级处理
```

```typescript
// ✅ 推荐的结构化输出防御模式
const SuggestionSchema = z.object({
  suggestions: z.array(z.object({
    templateId: z.number().describe('必须是模板列表中存在的 id'),
    // ...
  })).describe('匹配到的加分项列表，无匹配时为空数组 []'),
})

// prompt 中对应：
`请以如下 JSON 格式输出结果，不要输出其他内容：
{
  "suggestions": [
    { "templateId": 数字, ... }
  ]
}
无匹配时 suggestions 为 []`
```

---

### Zod 解析失败的通用兜底策略

建议在 `analyzeAndMatchNode` 里加一层降级处理：

```typescript
// 如果 withStructuredOutput 解析失败，降级到手动 JSON 解析
try {
  const result = await model.invoke([...])
  return { checkResults: result.suggestions.map(s => JSON.stringify(s)) }
} catch (err) {
  console.error('[analyzeAndMatch] 结构化输出失败，尝试降级解析:', err)
  // 降级：直接调用普通模型，手动解析 JSON
  const raw = await createChatModel(0).invoke([...])
  try {
    const text = String(raw.content)
    const jsonStr = text.match(/\{[\s\S]*\}/)?.[0] ?? '{}'
    const obj = JSON.parse(jsonStr)
    const sug = Array.isArray(obj.suggestions) ? obj.suggestions : []
    return { checkResults: sug.map((s: any) => JSON.stringify(s)) }
  } catch {
    return { checkResults: [] }
  }
}
```

## 三、LLM Think 模式的使用

### 什么是 Think 模式

Qwen3 系列模型支持"深度思考"（thinking）模式。开启后模型会先在 `<think>...</think>` 标签内进行推理，再给出最终答案。

- **优点**：推理更准确，适合复杂的匹配分析任务
- **缺点**：响应时间明显变长（通常 2~5 倍），token 消耗更多

### 当前默认关闭的原因

```typescript
// src/model.ts - 当前默认关闭 thinking
modelKwargs: { enable_thinking: false },
```

关闭原因：SSE 流式输出时，think 内容会作为 token 流推给前端，导致前端显示大量推理过程。

### 分场景使用 Think 模式

**场景一：`analyzeAndMatchNode` 开启思考（推荐）**

材料分析匹配是最关键的推理步骤，适合开启 thinking。但要注意过滤 think 内容。

```typescript
// src/model.ts 新增：
export function createThinkingModel() {
  return new ChatOpenAI({
    apiKey: getApiKey(),
    configuration: { baseURL: getBaseUrl() },
    modelName: getChatModel(),
    temperature: 1,                               // thinking 模式要求 temperature=1
    modelKwargs: { enable_thinking: true },
  })
}
```

```typescript
// src/nodes/applyNodes.ts - analyzeAndMatchNode 改用 thinking 模型
import { createThinkingModel } from '../model.js'

export async function analyzeAndMatchNode(state: ApplyStateType) {
  const model = createThinkingModel().withStructuredOutput(SuggestionSchema)
  // 其余不变...
}
```

**场景二：流式输出时过滤 think token**

think 模式下，`on_chat_model_stream` 事件会同时包含 thinking token 和正式 token。Qwen 通过 `reasoning_content` 字段区分：

```typescript
// src/service.ts - streamAgent 中过滤 think 内容
for await (const event of eventStream) {
  if (event.event === 'on_chat_model_stream') {
    const node = event.metadata?.langgraph_node
    if (node && SKIP_NODES.has(node)) continue

    const chunk = event.data?.chunk
    // 过滤推理内容（thinking token），只输出正式回答
    const token = chunk?.content           // 正式回答内容
    const think = chunk?.additional_kwargs?.reasoning_content  // 推理过程

    if (think) {
      // 可以选择不输出，或者以特殊类型推给前端展示
      // yield { type: 'thinking', data: { content: think } }
      continue
    }
    if (token) yield { type: 'token', data: { content: token } }
  }
}
```

**场景三：`consultNode` 保持关闭**

咨询回答节点需要快速响应，且答案来自知识库检索，不需要深度推理，保持 `enable_thinking: false`。

### Think 模式开启建议

| 节点               | Think 模式 | 理由                               |
|--------------------|------------|----------------------------------  |
| `classifyNode`     | ❌ 关闭    | 简单分类，无需深思，追求低延迟       |
| `askForMoreNode`   | ❌ 关闭    | 追问生成，语言任务，无需推理         |
| `analyzeAndMatchNode` | ✅ 开启 | 核心匹配推理，开启可减少误匹配       |
| `answerNode`       | ❌ 关闭    | RAG 回答，内容由知识库决定，无需推理 |

## 四、其他优化建议

### 4.1 RAG 存储升级（当前 MemoryVectorStore 的限制）

当前使用 `MemoryVectorStore`：
- ❌ 每次重启都需重新向量化所有文件
- ❌ 删除文件需要重建整个 Store（效率低）
- ❌ 无持久化，大量文件时内存压力大

**建议升级方向：**

```
方案 A：Chroma（本地持久化，Docker 一键启动）
  优点：支持按 metadata 过滤删除，持久化，性能好
  成本：需要 Docker + 多一个进程

方案 B：SQLite-vec（纯文件，零依赖）
  优点：无需额外服务，与现有 SQLite 搭配
  成本：功能较基础，无复杂过滤

当前项目建议：先用 Chroma，Docker 一行启动
```

```bash
docker run -d -p 8000:8000 chromadb/chroma
```

```typescript
// src/rag.ts 改用 Chroma
import { Chroma } from '@langchain/community/vectorstores/chroma'

const vectorStore = new Chroma(createEmbeddings(), {
  collectionName: 'id-ai-knowledge',
  url: 'http://localhost:8000',
})
// 支持按 metadata 删除：
await vectorStore.delete({ filter: { sourceFile: fileName } })
```

## 优化优先级建议

| 优先级 | 项目                              | 收益           | 成本   |
|--------|-----------------------------------|----------------|--------|
| P0     | Zod Schema 与后端 DTO 完全对齐    | 避免数据丢失   | 低     |
| P0     | analyzeAndMatchNode 开启 thinking  | 提升匹配准确率 | 低     |
| P1     | Zod 降级兜底处理                  | 增强健壮性     | 低     |
| P2     | RAG 升级为 Chroma                 | 性能+持久化    | 中     |
| P3     | 请求链路追踪                      | 易于排错       | 低     |
| P3     | Checkpoint 定期清理               | 控制磁盘占用   | 低     |
